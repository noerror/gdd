# 2. GDD 문서 구성

- 개요 문서 : 모든 컨텐츠를 포함하는 상위 지도 문서
- 세부 문서 : 각 세부 문서

## 2-1. 개요 문서

- **목적**: **단일 진실 소스(SSoT)**의 상위 지도, **계층 3단 이내**
- **구성**: 전체 구성을 한 눈에 볼 수 있으며 포함된 내용도 파악할 수 있도록 작성한다
- **분량**: 각 시스템 문서의 계층이 3단이 넘지 않도록 수평적으로 구성한다.
  - 예시: `400. 세계관 설정`대신  `401. 캐릭터 설정, 402. 배경 설정` 식으로 하위 문서가 너무 많은 내용을 포함해야하지 않도록 세분화

### 하위 문서 인덱스
- 하위 문서는 두 자리 혹은 세자리의 인덱스로 네이밍해서 관리
  - 상위번호가 큰 카테고리가 되도록 구성해서 파일명으로 정렬이 가능하고 중간에 추가가 용이하도록
  - 카테고리내에 문서추가가 용이하고 문서들을 그룹화해서 인식할 수 있다.
- 예시(참고용): 프로젝트 규모·조직 구조에 맞춰 조정하고, 그대로 사용하지 않는다.

  | 범위 | 분류 | 용도 |
  |---|---|---|
  | **000-009** | 마스터/인덱스 | 전체 개요, 네비게이션 |
  | **100-199** | 비전/컨셉 | 게임의 본질적 정체성 |
  | **200-299** | 시장/타깃 | 누구를 위한 게임인가 |
  | **300-399** | 코어 경험 | 무엇을 하며 왜 재미있는가 |
  | **400-499** | 세계관/내러티브 | 어떤 세계에서 펼쳐지는가 |
  | **500-599** | 시스템 상세 | 어떻게 작동하는가 |
  | **600-699** | 경제/진행 | 성장과 보상 구조 |
  | **700-799** | UI/UX | 사용자 인터페이스 |
  | **800-899** | 기술 | 구현 및 제약사항 |
  | **900-999** | 관리 | 검증, 리스크, 로드맵 |

## 3. 컨텐츠 구성

### 3-1. 비전/컨셉 문서

**특징:** 영감과 방향성을 제시하는 문서

**작성 시 주의사항:**

- **추상적이되 공허하지 않게**: "재미있는 게임"(X) → "극적인 역전의 카타르시스"(O)
- **의도가 담긴 무드보드**: 이미지는 많을수록 좋지만, 단순 나열이 아닌 각 이미지마다 "왜 이것인가"에 대한 선정 이유와 적용 의도를 설명합니다.
- **살아있는 디자인 필러**: 단순한 문구가 아니라, 실제 개발 중 마주치는 선택의 순간에 **의사결정의 기준(잣대)**으로 작동할 만큼 구체적이어야 합니다.

**작성 가이드**
- [게임 비전·컨셉 & 타깃/포지셔닝 작성 가이드](100-vision_concept_positioning.md) : 게임이 무엇을 약속하는지, 누구를 위해 만드는지, 포지션까지 한 번에.
- [코어 경험 & 코어 루프 작성 가이드](101-core_experience_loop.md) : 플레이어가 반복해서 겪는 경험과 루프 구조를 어떻게 정의하는지.

- [무드보드 & 레퍼런스 작성 가이드](103-moodboard_reference.md) : 이미지/레퍼런스 수집과 "왜 이걸 

### 3-2. 세계관/월드빌딩

**특징:** 게임의 배경이 되는 세계의 규칙, 역사, 이야기를 정의하여 플레이어 몰입의 토대를 마련하는 문서

**작성 시 주의사항:**

- **플레이와 연결:** 모든 설정은 게임 내 시스템이나 콘텐츠로 경험할 수 있어야 합니다. (단순 텍스트 나열 지양)
- **내적 일관성:** 역사, 지리, 문화, 기술 수준 등이 상호 모순 없이 논리적으로 연결되어야 합니다.
- **전달 방식 명시:** 방대한 설정을 플레이어에게 어떻게 전달할지(환경 스토리텔링, 아이템 설명, 대사 등) 구체화합니다.

**오디오/비주얼 디렉션**

- [오디오 디렉션 & 사운드스케이프 작성 가이드](181-audio_direction_soundscape.md) : 배경음, 효과음, UI 사운드, VO 톤, 환경 사운드 레이어링.

### 3-3. 시스템 문서

**특징:** 실행 가능한 구현 사양

**작성 시 주의사항:**

- **규칙의 수식화**: 모든 게임 메카닉은 단순 서술이 아닌 구체적인 수식으로 표현합니다. (예: `Damage = (Atk - Def) * Multiplier`)
- **상태와 흐름의 명확화**: 입력/출력/내부 상태 변화를 명확히 구분하고, 예외 케이스를 포함한 플로우차트를 작성합니다.
- **범위 기반 밸런스**: 단일 수치 대신 목표 범위를 제시하여 튜닝의 유연성을 확보합니다. (예: "승률 50%" (X) → "승률 45~55% 유지" (O))

### 3-4. 경제/밸런스 문서

**특징:** 장기 성장 곡선과 자원 흐름을 정의

**작성 시 주의사항:**

- **시뮬레이션 검증**: 모든 경제 모델은 스프레드시트 시뮬레이션을 통해 검증된 수치여야 합니다.
- **자원 흐름의 시각화**: 획득(Source)과 소비(Sink)의 균형을 다이어그램으로 시각화하여 경제 순환을 증명합니다.
- **시간 기반 진행**: 플레이어의 성장 곡선을 타깃 플레이 시간과 연동하여 콘텐츠 소모 속도를 제어합니다.
- **인플레이션 제어**: 재화 가치 하락에 대한 방어 기제(소각처 등)를 반드시 포함합니다.

**경제/밸런스 시스템**
