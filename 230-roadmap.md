# 로드맵 작성 가이드

## 1. 목적 (Purpose)
게임의 개발 및 운영 로드맵을 체계적으로 작성하여 팀 전체가 동일한 비전을 공유하고 효율적으로 협업하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: 마일스톤, 기능 우선순위, 일정 계획, 리스크 관리
- **제외**: 구체적인 작업 할당, 예산 계획, 마케팅 전략

## 3. 설계 목표 (Design Goals)

1. **명확한 비전 (Clear Vision)**: 팀 전체가 게임의 방향성을 이해합니다.
2. **현실적인 일정 (Realistic Schedule)**: 달성 가능한 마일스톤을 설정합니다.
3. **유연성 (Flexibility)**: 변화에 대응할 수 있는 여유를 둡니다.
4. **투명성 (Transparency)**: 진행 상황을 팀과 공유합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 마일스톤 (Milestones)
- **프리 프로덕션**: 프로토타입, 핵심 메커닉
- **프로덕션**: 콘텐츠 제작, 밸런싱
- **폴리싱**: 버그 수정, 최적화
- **출시 후**: 라이브옵스, 업데이트

### 4.2 기능 우선순위 (Feature Priority)
- **필수 기능**: MVP에 포함
- **중요 기능**: 출시 전 포함
- **선택 기능**: 출시 후 추가

### 4.3 리스크 & 대응 (Risks & Mitigation)
- **기술적 리스크**: 성능, 호환성
- **일정 리스크**: 지연 가능성

### 4.4 선택 및 심화 (Optional & Advanced)
- **버퍼 시간 확보**: 각 마일스톤에 10-20% 버퍼 시간 추가, 예상치 못한 문제 대응, 과거 프로젝트 데이터 기반 추정
- **애자일 vs 워터폴**: 인디/소규모는 애자일 권장(2주 스프린트), AAA/대규모는 하이브리드 접근, 마일스톤은 고정하되 스프린트는 유연하게
- **의존성 관리**: 크리티컬 패스 파악(핵심 시스템 우선 개발), 병렬 작업 가능 영역 식별, 팀 간 의존성 최소화
- **로드맵 커뮤니케이션**: 팀 내부 상세 로드맵 vs 외부 공개용 간략 로드맵 구분, 퍼블리셔/투자자용 버전 별도 작성, 정기 업데이트(월간)
- **변경 관리 프로세스**: 기능 추가/변경 시 팀 승인 필요, 우선순위 재평가, 일정 영향도 분석, 스코프 크립 방지

## 5. 관련 개념 (Key Concepts)

### 5.1 로드맵 구조
- **Milestone**: 주요 개발 단계 (프리프로덕션, 프로덕션, 폴리싱, 출시)
- **MVP (Minimum Viable Product)**: 최소 기능 제품
- **Critical Path**: 프로젝트 완료에 필수적인 작업 경로

### 5.2 개발 방법론
- **Agile**: 반복적 개발 방법론 (스프린트, 스크럼)
- **Waterfall**: 순차적 개발 방법론
- **Sprint**: 애자일의 2-4주 개발 주기

### 5.3 리스크 & 관리
- **Feature Creep**: 기능 추가로 인한 일정 지연 및 스코프 확대
- **Risk Management**: 리스크 식별, 평가, 대응 계획 수립
- **Scope Creep**: 프로젝트 범위가 통제 없이 확대되는 현상

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 마일스톤
- [ ] 모든 마일스톤이 명확하게 정의되었는가?
- [ ] 마일스톤 간 의존성이 고려되었는가?
- [ ] 각 마일스톤의 완료 기준이 명확한가?

### 6.2 우선순위
- [ ] 필수·중요·선택 기능이 구분되었는가?
- [ ] 기능 우선순위가 팀 전체에 공유되었는가?
- [ ] Feature Creep을 방지하는 메커니즘이 있는가?

### 6.3 리스크
- [ ] 주요 리스크가 식별되었는가?
- [ ] 각 리스크에 대한 대응 계획이 있는가?
- [ ] 리스크가 정기적으로 검토되는가?

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 마일스톤**
- **좋은 예 (Good)**: "M1 (3개월): 프로토타입 완성 / M2 (6개월): 핵심 콘텐츠 50% / M3 (9개월): 알파 테스트 / M4 (12개월): 출시, 명확한 기준과 일정"
- **안티패턴 (Bad)**: "마일스톤이 모호하거나 비현실적으로 짧음"

**Case 2: 기능 우선순위**
- **좋은 예 (Good)**: "필수: 핵심 전투 시스템 / 중요: 스토리 모드 / 선택: 멀티플레이, 우선순위 명확하여 리소스 집중"
- **안티패턴 (Bad)**: "모든 기능을 동시에 개발하려다 일정 지연"

**Case 3: 리스크 관리**
- **좋은 예 (Good)**: "리스크: 성능 문제 / 대응: 조기 최적화 테스트, 리스크: 일정 지연 / 대응: 버퍼 2주 확보"
- **안티패턴 (Bad)**: "리스크 식별 없이 진행하다 문제 발생 시 대응 불가"

**Case 4: Diablo IV - 라이브 로드맵과 세부 계획**
- **공개 로드맵 구조**: Quarterly 로드맵(분기별 3개월 계획 공개) + 월간 개발자 업데이트(상세 진행률), 예: "Q3 2024: 신규 클래스 50% 완성 / 보스 밸런스 패치 / 던전 2개 추가"
- **마일스톤 예시**: "Season 1 (5월~8월): 시즌 콘텐츠 50%, 베타 테스트 6주 → Season 1 출시 (8월) → Season 2 준비 (9월~)"
- **리스크 관리**: 기술 리스크(새 엔진 기능) → 조기 프로토타입(M2에서 검증), 내용 리스크(밸런스 불확실성) → 1개월 버퍼 확보, 일정 리스크 → 5%~10% 여유 시간 추가
- **투명한 지연 공지**: 예상 지연 시 4주 전 공지, 예: "보스 AI 복잡도로 인해 Season 2 출시 9월 → 10월로 연기", 지연 이유와 보상 아이템 동시 공지

**Case 5: Final Fantasy XIV - 확정적 로드맵과 신뢰 구축**
- **5년 로드맵 공개**: 매년 "Letter from the Producer" 발표(향후 12개월 계획), 예: "2024년: 새 익스팬션 개발 50% 완성 / 레이드 4층 + 알라이언스 레이드 1개 / 주거 시스템 확장"
- **상세한 마일스톤**: Patch 6.4 (12월) → Patch 6.5 (2월) → Patch 7.0 (Early Access 7월, 정식 8월), 각 패치마다 명확한 콘텐츠(레이드 1층, 던전 1개, 새 직업 스킬 트리)
- **우선순위 공개**: "우리가 할 수 있는 것" vs "우리가 하고 싶지만 할 수 없는 것" 구분, 예: "플레이어 주택 확장 (할 수 있음) vs 대규모 주택 시스템 재설계 (일정상 불가능)"
- **평가**: 15년 운영 게임이지만 D30 리텐션 70% 유지, 신뢰도 93% (게이머 설문)

**Case 6: Genshin Impact - 민첩한 로드맵과 즉각적 조정**
- **공개 로드맵**: 분기별 공개(3개월 계획), 월간 업데이트(진행 상황), 예: "2024년 2분기: 신 지역 2개, 신 캐릭터 5명, 신 무기 4개, 메인 스토리 진행"
- **유연한 우선순위**: "플레이어 피드백 기반 즉각적 조정", 예: 신 캐릭터 Nahida 강도 과다 → 2주 내 너프 패치, 원래 계획된 6월 패치를 5월로 앞당김
- **마일스톤 버퍼**: 각 마일스톤 15%~20% 버퍼 시간 추가(예: 신 지역 개발 12주 예정 → 14.4주 예비 시간), 큰 문제 발생 시에도 다음 마일스톤 지연 최소화
- **결과**: 지난 3년간 마일스톤 달성율 98%, 평균 지연일수 2.5일, MAU 500만 이상 유지

**Case 7: No Man's Sky - 도전적 출시 후 로드맵 신뢰 회복**
- **로드맵 재수립**: 2016년 출시 후 비판(약속 기능 부재) → 2017년부터 공개 로드맵 게시(6개월 단위 계획 공개), "Beyond" 확장팩(2019), "Origins" 확장팩(2021), "Frontiers" 확장팩(2023)
- **마일스톤 계획**: 각 확장팩 마다 6-9개월 개발(예: Beyond 9개월), 분기별 마일스톤(M1: 코어 기능 50%, M2: 콘텐츠 50%, M3: 폴리싱 100%), 각 마일스톤 20% 버퍼
- **투명한 커뮤니케이션**: 매 월간 개발자 노트(진행률, 기술적 도전과제, 다음달 계획), 지연 발생 시 최소 4주 전 공지 + 지연 이유 상세 설명, 플레이어 피드백 반영(포럼 투표로 우선순위 결정)
- **신뢰 회복 결과**: 2016년 MAU 200만 → 비판 → 2019년 MAU 400만 → 2024년 MAU 750만으로 회복, 지난 5년간 평균 마일스톤 달성율 92%

**Case 8: Sea of Thieves - 계절 기반 로드맙과 플레이어 피드백 통합**
- **로드맵 구조**: 분기별 "시즌"(3개월), 각 시즌마다 공개 로드맵 공개(주요 기능, 밸런스 변화, 신규 콘텐츠), 예: "Season 1: 기지 건설 시스템, 신규 던전 2개, 신규 무기 라인 5개"
- **마일스톤과 우선순위**: M1(3주): 핵심 기능 프로토타입 → M2(6주): 콘텐츠 제작 50% → M3(6주): 콘텐츠 제작 100% + 폴리싱 → M4(2주): 버그 수정 및 준비, 각 마일스톤마다 5-10% 버퍼
- **플레이어 피드백 통합**: 월간 개발자 스트림에서 실시간 Q&A, 포럼 투표로 우선순위 결정, 플레이어 제안된 기능 중 상위 3개는 검토(최소 6개월 간격)
- **결과**: 시즌별 마일스톤 달성율 94%, 평균 지연 일수 5일, 분기별 리텐션 D30 68% 유지

**Case 9: The Elder Scrolls Online - 장기 로드맵과 대규모 확장팩 계획**
- **5년 로드맵 공개**: "The Elder Scrolls Online: Gates of Oblivion"(2021년) → "High Isle"(2022년) → "Gold Road"(2024년) 각 확장팩 마다 구체적 계획 선언
- **마일스톤 계획**: 각 확장팩 마다 12개월 개발 주기(M1: 3개월 코어 디자인, M2: 6개월 콘텐츠 제작, M3: 2개월 폴리싱, M4: 1개월 테스트), 각 마일스톤 15% 버퍼
- **투명한 지연 관리**: 확장팩 지연 시 3개월 전 공지(예: "Gold Road" 예정일 6월 → 8월로 연기), 지연 이유 공개 + 보상 아이템 사전 공지, 플레이어 불만 사전 차단
- **커뮤니티 피드백**: 공식 포럼에서 "Feature Voting" 분기마다 실시(차기 콘텐츠 우선순위 결정), 개발진이 플레이어 투표 결과 반영 의무, 투표 결과 공개
- **결과**: 19년 운영 게임, MAU 300만 유지, 리텐션 D30 65%, 각 확장팩 출시 후 2주간 DAU 500만 이상

## 8. 핵심 인사이트 (Key Insights)

- **현실적인 일정**: 과거 프로젝트 데이터를 기반으로 현실적인 일정을 수립하고, 버퍼를 확보하세요.
- **명확한 우선순위**: 필수·중요·선택 기능을 명확히 구분하여 리소스를 효율적으로 배분하세요.
- **리스크 관리**: 주요 리스크를 조기에 식별하고 대응 계획을 수립하세요.
- **정기 검토**: 로드맵을 정기적으로 검토하고 필요 시 조정하여 유연성을 유지하세요.
