# 로드맵 작성 가이드

## 1. 목적 (Purpose)
게임의 개발 및 운영 로드맵을 체계적으로 작성하여 팀 전체가 동일한 비전을 공유하고 효율적으로 협업하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: 마일스톤, 기능 우선순위, 일정 계획, 리스크 관리
- **제외**: 구체적인 작업 할당, 예산 계획, 마케팅 전략

## 3. 설계 목표 (Design Goals)

1. **명확한 비전 (Clear Vision)**: 팀 전체가 게임의 방향성을 이해합니다.
2. **현실적인 일정 (Realistic Schedule)**: 달성 가능한 마일스톤을 설정합니다.
3. **유연성 (Flexibility)**: 변화에 대응할 수 있는 여유를 둡니다.
4. **투명성 (Transparency)**: 진행 상황을 팀과 공유합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 마일스톤 (Milestones)
- **프리 프로덕션**: 프로토타입, 핵심 메커닉
- **프로덕션**: 콘텐츠 제작, 밸런싱
- **폴리싱**: 버그 수정, 최적화
- **출시 후**: 라이브옵스, 업데이트

### 4.2 기능 우선순위 (Feature Priority)
- **필수 기능**: MVP에 포함
- **중요 기능**: 출시 전 포함
- **선택 기능**: 출시 후 추가

### 4.3 리스크 & 대응 (Risks & Mitigation)
- **기술적 리스크**: 성능, 호환성
- **일정 리스크**: 지연 가능성

## 5. 관련 개념 (Key Concepts)

- **Milestone**: 주요 개발 단계
- **MVP (Minimum Viable Product)**: 최소 기능 제품
- **Feature Creep**: 기능 추가로 인한 일정 지연
- **Risk Management**: 리스크 식별 및 대응

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 마일스톤
- [ ] 모든 마일스톤이 명확하게 정의되었는가?
- [ ] 마일스톤 간 의존성이 고려되었는가?
- [ ] 각 마일스톤의 완료 기준이 명확한가?

### 6.2 우선순위
- [ ] 필수·중요·선택 기능이 구분되었는가?
- [ ] 기능 우선순위가 팀 전체에 공유되었는가?
- [ ] Feature Creep을 방지하는 메커니즘이 있는가?

### 6.3 리스크
- [ ] 주요 리스크가 식별되었는가?
- [ ] 각 리스크에 대한 대응 계획이 있는가?
- [ ] 리스크가 정기적으로 검토되는가?

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 마일스톤**
- **좋은 예 (Good)**: "M1 (3개월): 프로토타입 완성 / M2 (6개월): 핵심 콘텐츠 50% / M3 (9개월): 알파 테스트 / M4 (12개월): 출시, 명확한 기준과 일정"
- **안티패턴 (Bad)**: "마일스톤이 모호하거나 비현실적으로 짧음"

**Case 2: 기능 우선순위**
- **좋은 예 (Good)**: "필수: 핵심 전투 시스템 / 중요: 스토리 모드 / 선택: 멀티플레이, 우선순위 명확하여 리소스 집중"
- **안티패턴 (Bad)**: "모든 기능을 동시에 개발하려다 일정 지연"

**Case 3: 리스크 관리**
- **좋은 예 (Good)**: "리스크: 성능 문제 / 대응: 조기 최적화 테스트, 리스크: 일정 지연 / 대응: 버퍼 2주 확보"
- **안티패턴 (Bad)**: "리스크 식별 없이 진행하다 문제 발생 시 대응 불가"

## 8. 핵심 인사이트 (Key Insights)

- **현실적인 일정**: 과거 프로젝트 데이터를 기반으로 현실적인 일정을 수립하고, 버퍼를 확보하세요.
- **명확한 우선순위**: 필수·중요·선택 기능을 명확히 구분하여 리소스를 효율적으로 배분하세요.
- **리스크 관리**: 주요 리스크를 조기에 식별하고 대응 계획을 수립하세요.
- **정기 검토**: 로드맵을 정기적으로 검토하고 필요 시 조정하여 유연성을 유지하세요.
