# 매치메이킹·세션·네트워크 작성 가이드

## 1. 목적 (Purpose)
멀티플레이 게임의 매치메이킹, 세션 관리, 네트워크 요구사항을 체계적으로 정의하여 원활한 멀티플레이 경험을 제공하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: 매치메이킹 알고리즘, 세션 생성·관리, 네트워크 요구사항, 지연 처리
- **제외**: 서버 인프라 구축, 네트워크 프로그래밍 세부사항

## 3. 설계 목표 (Design Goals)

1. **빠른 매칭 (Quick Matching)**: 플레이어가 빠르게 게임에 참여할 수 있습니다.
2. **공정한 매칭 (Fair Matching)**: 실력이 비슷한 플레이어를 매칭합니다.
3. **안정적인 연결 (Stable Connection)**: 네트워크 지연과 끊김을 최소화합니다.
4. **확장성 (Scalability)**: 동시 접속자 증가에 대응합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 매치메이킹 (Matchmaking)
- **알고리즘**: MMR 기반, 대기 시간 고려
- **매칭 조건**: 지역, 핑, 스킬 레벨

### 4.2 세션 관리 (Session Management)
- **세션 생성**: 방 만들기, 초대
- **세션 유지**: 호스트 마이그레이션, 재연결

### 4.3 네트워크 (Network)
- **요구사항**: 최소 대역폭, 최대 핑
- **지연 처리**: 예측, 보간, 롤백

### 4.4 선택 및 심화 (Optional & Advanced)
- **매칭 로직**: 지역별(Region) 핑 제한 설정, 파티 vs 솔로 매칭 가중치 조정
- **예외 상황**: 매칭 서버 다운 시 오프라인 모드 전환, 매칭 취소 시 페널티 예외 처리
- **백필(Backfill)**: 난입 허용 구간 설정, 난입 유저에 대한 추가 보상 정책
- **봇 운영**: 신규·복귀 유저 첫 매치의 봇 비율, 실력 상승 시 봇 비율 축소 규칙, 결원 봇 백필, 봇 사용 고지·옵트아웃 정책

## 5. 관련 개념 (Key Concepts)

- **MMR (Matchmaking Rating)**: 플레이어 실력 지표
- **Ping**: 네트워크 지연 시간
- **Host Migration**: 호스트 변경 시 세션 유지
- **Lag Compensation**: 네트워크 지연 보정
- **버킷 곱셈 (Bucket Multiplication)**: 매칭 조건을 겹칠수록 가용 인구가 곱으로 축소되는 현상. 스킬 15% × 레이턴시 10% × 평판 5% = 매칭 가능 인구 0.08%
- **TrueSkill2**: 승패 외에 킬/데스·판수·파티 여부까지 반영하는 스킬 평가 모델. 승패만 세는 구모델은 수렴에 80~120경기 소요
- **Bot Backfill (봇 백필)**: 결원·이탈·인구 부족을 봇으로 메워 매치 성립을 보장하는 운영 기법. 대기 시간 단축과 로비 유지의 완충재
- **비동기 멀티플레이 (Asynchronous Multiplayer)**: 친구 대신 친구의 플레이어 모델과 대전하는 방식 (Forza의 Drivatar). 플레이어 모사 AI에는 악용 방지(리와인드 구간 학습 제외)와 민폐 행동 필터 레이어가 필수

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 매치메이킹
- [ ] 매치메이킹 알고리즘이 정의되었는가?
- [ ] 평균 대기 시간이 목표 내인가?
- [ ] 지역·핑·스킬 레벨이 고려되는가?
- [ ] 매칭 조건별 인구 축소율을 곱셈으로 계산해 검증했는가?
- [ ] 이탈(탈주) 원인 분석이 매치 공정성(팀 스킬 갭) 지표와 연결되어 있는가?
- [ ] 파티 vs 솔로 매칭 정책(분리 큐·대기 페널티·큐 인원 제한)이 정의되었는가?
- [ ] 게임 모드 수가 동시 접속 인구 대비 과다하지 않은가?
- [ ] 매칭 파라미터 실험 방법이 정의되었는가? (인구를 쪼개는 A/B 대신 시간 단위 전환 방식 권장)

### 6.2 세션
- [ ] 세션 생성·관리 프로세스가 명확한가?
- [ ] 호스트 마이그레이션이 지원되는가?
- [ ] 재연결 메커니즘이 있는가?
- [ ] 서버 다운 시 진행 손실 방지책(주기 저장·복구)이 테스트 빌드부터 구현되는가?
- [ ] 매칭 대기 중 플레이어가 할 것(연습장·미니게임 등)이 제공되는가?

### 6.3 네트워크
- [ ] 네트워크 요구사항이 명시되었는가?
- [ ] 지연 처리 메커니즘이 구현되었는가?
- [ ] 네트워크 상태 표시가 제공되는가?

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 매치메이킹**
- **좋은 예 (Good)**: "MMR ±100 범위 내 매칭, 30초 대기 후 범위 확대, 평균 대기 시간 1분 이내, 지역·핑 우선 고려"
- **안티패턴 (Bad)**: "완벽한 매칭만 추구하여 대기 시간 과도하게 길어짐"

**Case 2: 호스트 마이그레이션**
- **좋은 예 (Good)**: "호스트 이탈 시 자동으로 다음 플레이어에게 호스트 이전, 게임 중단 없이 계속 진행"
- **안티패턴 (Bad)**: "호스트 이탈 시 게임 종료로 플레이어 좌절"

**Case 3: 지연 처리**
- **좋은 예 (Good)**: "클라이언트 예측 + 서버 조정, 롤백 메커니즘으로 부정확한 예측 수정, 최대 핑 150ms 제한"
- **안티패턴 (Bad)**: "지연 처리 없이 네트워크 지연이 그대로 노출"

**Case 4: 매치메이킹 알고리즘 (Rocket League)**
- **좋은 예 (Good)**: "MMR ±50 범위로 시작, 30초마다 ±25씩 확대, 최대 3분 대기 시 ±200 범위, 지역 우선 매칭 후 글로벌 확장, 평균 대기 시간 45초"
- **안티패턴 (Bad)**: "고정 MMR 범위로 고랭크 대기 시간 10분 이상"

**Case 5: 크로스플레이와 호스트 마이그레이션 (Fortnite)**
- **좋은 예 (Good)**: "PC-콘솔-모바일 크로스플레이, 입력 방식별 매치풀 분리 옵션, 호스트 서버 방식으로 마이그레이션 불필요, 안정적 연결 유지"
- **안티패턴 (Bad)**: "P2P 방식에서 호스트 이탈 시 게임 종료"

**Case 6: 네트워크 상태 표시와 지연 보정 (Valorant)**
- **좋은 예 (Good)**: "실시간 핑 표시 (숫자 + 색상), 패킷 손실률 모니터링, 128-tick 서버로 정확도 향상, 서버측 히트 검증으로 치팅 방지"
- **안티패턴 (Bad)**: "네트워크 상태 정보 없이 플레이어가 지연 원인 파악 불가"

**Case 7: 이탈 원인 실측 (Halo 5)**
- **실측 결과**: 레이턴시·대기 시간·연패는 이탈과 거의 무관했고, 이탈을 부른 유일한 요인은 '그 판이 불공정했는가'. 팀 스킬 갭 100에 탈주율 2배, 개인 스킬 갭 관리만으로 탈주율 4배 감소
- **좋은 예 (Good)**: "TrueSkill2(킬/데스·판수·파티 반영)로 재측정하자 구모델에서 평평해 보이던 갭-탈주 상관이 뚜렷하게 드러남"
- **안티패턴 (Bad)**: "'우리 게임은 스킬 갭이 안 중요하더라'라는 결론 — 스킬 시스템이 무능해 갭을 측정하지 못했을 가능성을 배제한 판단"

**Case 8: P2P 선택의 라이브 비용 (For Honor)**
- **사건**: 저지연을 위해 호스트 없는 메시 P2P+결정론적 시뮬레이션을 채택했으나, 4v4에서 '가장 약한 링크가 전원을 망치는' 구조적 한계 발생. 출시 후 첫 달 100개 이상의 버그를 고쳐도 품질 유리천장에 막힘
- **결말**: 풀팀 8개월을 들여 전용 서버로 이전 — "작은 문제는 라이브에서 반드시 커진다"
- **교훈**: 네트워크 토폴로지 선택은 되돌리는 비용이 팀 단위·연 단위이므로, 최대 인원 모드 기준의 최악 시나리오로 검증 후 결정할 것

**Case 9: 신규 유저 첫 매치 봇 구성 (Fortnite / Gears of War)**
- **좋은 예 (Good)**: "Fortnite 챕터 2는 신규·복귀 유저의 첫 솔로 매치를 봇 위주로 구성해 사실상 튜토리얼로 활용, 스킬 매치메이킹과 연동해 실력이 오르면 봇 비율 축소. Gears of War는 첫 멀티 매치에 봇+데미지 보너스로 '첫 킬 경험'을 보장 (첫 게임 무킬 유저가 이탈한다는 데이터 기반)"
- **안티패턴 (Bad)**: "봇 투입 은폐 — 발각 시 그동안의 성취가 기만으로 변함 (Mario Kart Tour 사례). 고실력자 매치의 봇은 파밍 대상으로 악용됨"

**Case 10: 공정 매칭의 역효과 (Clash of Clans)**
- **사건**: 클랜전이 항상 동급 매칭이라 업그레이드가 제로섬이 됨 — 최적 전략이 '낮은 타운홀에서 만렙 찍고 성장 정지'+위장 기지로 매칭 알고리즘 속이기가 됨
- **해결**: 매칭 공정성을 버린 별도 리그(시드 후 무작위 대전) 신설
- **교훈**: 매칭 공정성은 목적이 아니라 수단이며, 성장 동기와 충돌하면 모드별로 다른 매칭 철학을 적용해야 함

## 8. 핵심 인사이트 (Key Insights)

### 8.1 매칭 공정성과 인구 산수
- **빠른 매칭과 공정성 균형**: 완벽한 매칭을 추구하기보다 대기 시간과 매칭 품질의 균형을 맞추세요.
- **이탈을 부르는 것은 불공정한 매치뿐 (Halo 5 실측)**: 핑·대기 시간·연패는 이탈과 거의 무관했고 연패 끊기(losing streak breaker)도 근거가 없었습니다. 매치 공정성(팀·개인 스킬 갭)을 이탈 관리의 1순위 지표로 삼으세요.
- **매칭 조건은 곱셈으로 인구를 태운다**: 동접 10만도 플랫폼 분리→매치 중 인구 제외→모드 5개 분할을 거치면 분당 가용 100명대로 줄어듭니다. 조건을 매치메이커에 다 던지지 말고 데이터로 선별한 소수 조건만 거세요. 우선순위의 현실은 핑 > 큐 시간 > 매치 품질이며, 유저의 "더 기다려도 좋다"는 자기보고는 데이터와 다릅니다(수 분 초과 시 이탈).
- **풀스택 파티 1팀이 시간당 최대 20명을 쫓아낸다 (Halo 5)**: 4명이 놀 권리 vs 시간당 20명 이탈의 산수는 성립하지 않습니다. 공정 매치만 허용(스택은 장시간 대기), 솔로/듀오 큐 분리(LoL), 최상위 구간 2인 큐 제한(Overwatch) — 완벽한 해법은 없고 전부 트레이드오프이므로 정책을 명시적으로 선택하세요.
- **매치메이킹에는 표준 A/B가 깨진다**: 최적 매칭 파라미터는 전체 인구 구성의 함수인데 A/B는 인구를 쪼개므로 잘못된 모집단의 최적값을 학습합니다. 매시간 설정을 통째로 전환해 같은 인구에서 비교하는 시간 단위 전환 실험으로 대체하세요.

### 8.2 세션·네트워크·봇 운영
- **호스트 마이그레이션**: 호스트 이탈 시에도 게임이 계속되도록 호스트 마이그레이션을 지원하세요.
- **지연 처리**: 클라이언트 예측과 서버 조정으로 네트워크 지연을 최소화하세요.
- **네트워크 상태 표시**: 플레이어에게 현재 핑과 네트워크 상태를 표시하여 투명성을 제공하세요.
- **봇 운영 3원칙: 고지하라, 옵트아웃을 줘라, 실력에 따라 빼라**: 은폐는 발각 시 기만이 되고, 고실력 매치의 봇은 파밍 대상이 됩니다. 봇 난이도 스펙트럼은 기대 실력에 도달할 수 없는 플레이어의 참여 수단(접근성)이기도 합니다. 인간 흉내 봇의 반응 스펙은 최소 지연이 아니라 인간 반응 시간(200~250ms)에 맞추세요 — Halo Infinite는 이를 봇 8기의 총 의사결정 예산(봇당 약 31ms)으로 썼습니다.
- **래더 보상 밀도를 설계하라**: 티어 승급은 몇 달에 한 번이라 보상 밀도가 재앙적입니다. 50점 단위 마이크로 보상을 랭크가 떨어져도 회수하지 않는 형태로 지급하고, 배치를 높게 통과한 플레이어에게는 건너뛴 보상을 일괄 지급하세요.
- **크로스 서버 진행 공유는 런칭 혼잡기의 안전판**: 캐릭터가 서버 귀속이 아니면 유저가 스스로 한산한 타 지역 서버로 우회해 이탈하지 않습니다. 반대로 모드별 큐 시간 격차(예: 특정 모드만 수 시간)는 그 콘텐츠를 죽이므로, 베타에서는 인구를 소수 모드에 집중시키세요.
