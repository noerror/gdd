# 로컬라이제이션 & 컬처럴라이제이션 작성 가이드

## 1. 목적 (Purpose)
게임의 로컬라이제이션(L10N)과 컬처럴라이제이션 요구사항을 설계 단계부터 체계적으로 정의하여, 글로벌 출시 시 번역 품질·UI 재작업·문화적 논란·규제 리스크를 최소화하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: 대상 언어·시장 선정, 텍스트 파이프라인(스트링 키·플레이스홀더·복수형 처리), UI 확장성(텍스트 팽창률·폰트/문자셋·RTL), 컬처럴라이제이션(문화적 금기·상징·색상), VO 로컬라이제이션 전략(자막 vs 더빙), 지역별 등급·규제 대응, LQA 계획
- **제외**: 원문 대사·시나리오 작성, 번역 실무(개별 번역문 산출), 폰트 렌더링 기술 구현
- **문서 경계**:
  - 원문 대사·연출 자체는 [대사·연출 & 캐릭터 보이스](163-dialogue_cinematic_vo.md)가 소유합니다. 이 문서는 그 결과물을 다국어로 옮기는 전략만 다룹니다.
  - 폰트·타이포그래피 스타일 정의는 [UI 레이아웃 & 화면 구조](172-ui_layout_screens.md)가 소유합니다. 이 문서는 언어별 문자셋·확장성 요구사항만 정의합니다.
  - 번역 관리 툴 체인(CAT 툴·스트링 테이블 빌드 연동)은 [툴·파이프라인 요구사항](202-tools_pipeline_requirements.md)이 소유합니다.

## 3. 설계 목표 (Design Goals)

1. **확장성 (Scalability)**: 신규 언어 추가가 코드 수정 없이 데이터 추가만으로 가능해야 합니다.
2. **일관성 (Consistency)**: 용어집(Glossary)과 번역 메모리(TM)로 모든 언어에서 용어와 톤을 통일합니다.
3. **문화적 적합성 (Cultural Fit)**: 단순 번역을 넘어 각 시장의 문화적 맥락에 맞게 콘텐츠를 적응시킵니다.
4. **규제 준수 (Compliance)**: 지역별 심의 등급·확률 공시·표현 규제를 출시 전에 충족합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 대상 언어·시장 선정 (Target Languages & Markets)
- **언어 티어**: 1차(출시 필수: 예, 영어·한국어·일본어), 2차(출시 후 추가: 예, FIGS - 프랑스어/이탈리아어/독일어/스페인어), 3차(성과 기반 검토)
- **선정 기준**: 시장 규모, 장르 선호도, 플랫폼별 매출 비중, 번역·LQA 비용 대비 기대 수익
- **범위 수준 정의**: 언어별로 텍스트만/자막만/풀더빙 중 어느 수준까지 지원하는지 명시

### 4.2 텍스트 파이프라인 (Text Pipeline)
- **스트링 키 원칙**: 모든 노출 텍스트는 스트링 테이블 키로 참조하며 코드·에셋에 하드코딩 금지
- **플레이스홀더**: 변수 삽입은 `{player_name}` 같은 명명된 플레이스홀더 사용, 어순이 언어마다 다르므로 위치 기반(`%s %d`) 조합 금지
- **복수형 처리**: 언어별 복수형 규칙 차이(영어 2형, 러시아어 3형, 아랍어 6형, 한국어·일본어 1형)를 지원하는 플루럴 키 구조 정의
- **컨텍스트 제공**: 각 스트링에 화자·상황·글자 수 제한을 메타데이터로 첨부하여 컨텍스트 부족 문제 예방

스트링 테이블 항목 데이터 템플릿 예시:

```json
{
  "key": "quest_101_greeting_blacksmith",
  "source": "Sharpen your blade, {player_name}. You'll need it.",
  "context": "대장장이 NPC가 첫 퀘스트 수락 직후 건네는 인사. 거칠지만 호의적인 톤.",
  "char_limit": 60,
  "speaker": "npc_blacksmith_grom",
  "plural_forms": null,
  "screenshot_ref": "loc/screens/quest_101_greeting.png"
}
```

### 4.3 UI 확장성 & 문자셋 (UI Expansion & Character Sets)
- **팽창률 예산**: 영어 기준 UI 요소별로 팽창 여유 공간을 확보하고, 초과 시 자동 축소·말줄임·툴팁 전환 규칙 정의
- **폰트/문자셋**: 언어별 필요 글리프 범위와 폴백(Fallback) 폰트 체계 요구사항 정의
- **RTL 대응**: 아랍어·히브리어 지원 시 텍스트 방향뿐 아니라 UI 레이아웃 미러링 범위(진행 바, 아이콘 방향 등) 명시

언어별 통상 팽창률·문자셋 요구 예시:

| 언어 | 통상 팽창률 (영어 기준) | 문자셋/폰트 요구 | 주의사항 |
|---|---|---|---|
| 독일어 | 통상 +20~35% | 라틴 확장 (움라우트) | 복합 명사로 단어 하나가 매우 길어짐 |
| 프랑스어 | 통상 +15~25% | 라틴 확장 (악상 부호) | 구두점 앞 공백 규칙 |
| 러시아어 | 통상 +10~20% | 키릴 문자 | 복수형 3형 처리 필요 |
| 한국어 | 통상 -10~0% | 한글 (완성형 11,172자 권장) | 조사 처리(을/를, 이/가), 단어 단위 줄바꿈 |
| 일본어 | 통상 -10~-40% | 한자+가나 수천 자 | 폰트 용량 큼, 금칙 문자 줄바꿈 규칙 |
| 중국어(간체) | 통상 -20~-50% | 간체 한자 수천 자 | 획이 많아 최소 폰트 크기 확보 필수 |
| 아랍어 | 통상 +20~25% | RTL, 문맥형 연결 글리프 | UI 미러링, 숫자는 LTR 혼용 |

### 4.4 컬처럴라이제이션 & 지역 규제 (Culturalization & Regional Compliance)
- **문화적 금기 검수**: 종교 상징, 손동작·제스처, 역사·정치적 상징, 동물·숫자의 문화적 의미를 시장별로 사전 검수
- **색상·상징 의미**: 색상의 문화별 함의(예: 흰색 - 서구권 순수 vs 동아시아 일부 문화권 장례) 차이를 아트·UI에 반영
- **지역별 등급·규제**: 시장별 심의 기관(GRAC·CERO·ESRB·PEGI·USK) 등급 목표, 확률형 아이템 확률 공시 의무(한국·중국 등), 특정 표현물 규제(폭력·도박·상징물) 대응 계획
- **지역별 콘텐츠 변형**: 규제·문화 대응을 위한 지역별 아트·연출 변형 목록과 관리 방식

### 4.5 VO 로컬라이제이션 & LQA (VO Localization & LQA)
- **자막 vs 더빙**: 시장별 선호(통상 독일·프랑스는 더빙 선호가 강하고, 북유럽은 자막 수용도가 높음)와 비용을 고려한 언어별 VO 전략 매트릭스
- **더빙 제약**: 립싱크 수준(정밀/타임싱크/자유), 대사 길이 제약, 스트링 프리즈 이후 대본 변경 금지 원칙
- **LQA 계획**: 언어별 인게임 검수(텍스트 잘림·오역·컨텍스트 오류·문화적 부적절성) 라운드 수, 원어민 테스터 확보 방안, 버그 분류 체계

### 4.6 선택 및 심화 (Optional & Advanced)
- **의사 로컬라이제이션 빌드**: 개발 초기부터 pseudo-loc 빌드를 CI에 포함하여 하드코딩·잘림·문자셋 문제를 조기 감지
- **TM·용어집 운영**: 번역 메모리 재활용률 추적, 시리즈·업데이트 간 용어 일관성 유지 프로세스
- **커뮤니티 번역**: 공식 미지원 언어의 커뮤니티 번역 허용 정책과 도구 지원 범위
- **문화 자문**: 주요 시장별 문화 컨설턴트 검수 단계 도입(스토리·아트 마일스톤별)

## 5. 관련 개념 (Key Concepts)

- **i18n vs L10N vs Culturalization**: 국제화(i18n)는 다국어를 담을 수 있는 기술 구조 설계, 로컬라이제이션(L10N)은 특정 언어·지역으로의 변환, 컬처럴라이제이션은 문화적 맥락까지 적응시키는 상위 개념
- **Pseudo-localization**: 실제 번역 전에 가짜 확장 문자열([!!! Šħàŕpéñ ýöúŕ ßļàdé !!!])로 UI 잘림·하드코딩·인코딩 문제를 검출하는 기법
- **String Freeze**: 특정 시점 이후 원문 텍스트 변경을 금지하여 번역·녹음 재작업을 방지하는 마일스톤
- **Translation Memory (TM)**: 과거 번역문을 재활용하는 데이터베이스로, 비용 절감과 용어 일관성의 핵심
- **컨텍스트 부족 문제**: 번역가가 화자·상황·UI 위치를 모른 채 스트링만 받으면 오역이 발생하는 문제. 스크린샷·메타데이터 제공으로 예방
- **LQA (Linguistic Quality Assurance)**: 번역문을 실제 게임 화면에서 검증하는 언어 품질 검수 단계

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 텍스트 파이프라인
- [ ] 모든 노출 텍스트가 스트링 키로 관리되고 하드코딩이 없는가?
- [ ] 플레이스홀더가 명명식이며 어순 변경에 대응 가능한가?
- [ ] 복수형·성별 변형 등 언어별 문법 규칙이 지원되는가?
- [ ] 각 스트링에 컨텍스트(화자·상황·글자 수 제한)가 첨부되는가?

### 6.2 UI & 컬처럴라이제이션
- [ ] UI 요소별 텍스트 팽창 여유와 초과 시 처리 규칙이 정의되었는가?
- [ ] 대상 언어의 문자셋·폴백 폰트 요구사항이 정의되었는가?
- [ ] RTL 지원 여부와 미러링 범위가 결정되었는가?
- [ ] 문화적 금기·상징·색상 검수 프로세스가 계획되었는가?
- [ ] 텍스트 강조 연출(크기·색·애니메이션)을 언어별 번역자가 스크립트로 직접 조정할 수 있는가?
- [ ] 휴대 모드·소형 화면에서 언어별 최소 폰트 크기가 검증되고, 폰트 크기 옵션이 제공되는가?

### 6.3 VO·규제·LQA
- [ ] 언어별 자막/더빙 수준이 비용과 함께 결정되었는가?
- [ ] 스트링 프리즈 시점이 마일스톤에 반영되었는가?
- [ ] 대상 시장별 심의 등급 목표와 확률 공시 의무가 파악되었는가?
- [ ] 언어별 LQA 라운드와 원어민 테스터 확보 방안이 있는가?

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 스트링 키 관리와 하드코딩**
- **좋은 예 (Good)**: "프로토타입 단계부터 모든 텍스트를 스트링 테이블로 관리하고, pseudo-loc 빌드를 주간 CI에 포함하여 하드코딩된 텍스트를 자동 검출. 번역 시작 시점에 추가 엔지니어링 작업 0건"
- **안티패턴 (Bad)**: "UI 프리팹과 코드 곳곳에 영어 텍스트가 하드코딩되어, 로컬라이제이션 착수 시점에 수천 개 텍스트를 수작업으로 추출·교체하는 재작업 발생"

**Case 2: UI 텍스트 팽창 대응**
- **좋은 예 (Good)**: "버튼·툴팁 등 UI 요소별로 영어 기준 +35% 여유 폭을 설계 규칙으로 명시하고, 초과 시 자동 폰트 축소(최소 크기 하한 포함) → 말줄임 순의 폴백 규칙 정의"
- **안티패턴 (Bad)**: "영어 텍스트에 딱 맞춘 고정 폭 버튼으로 제작하여 독일어 빌드에서 버튼 텍스트가 잘리거나 겹침. LQA에서 발견된 잘림 버그가 수백 건으로 UI 전면 수정"

**Case 3: 다국어 폰트·문자셋 통합 (The Legend of Zelda: Breath of the Wild)**
- **좋은 예 (Good)**: "다수의 텍스트 언어와 복수의 음성 언어를 단일 빌드에 통합하고, 라틴·키릴·한중일 문자셋 각각에 게임의 아트 톤과 어울리는 폰트를 적용. 언어별로 UI 가독성을 개별 검증하여 어떤 언어로 플레이해도 동일한 완성도 제공"
- **안티패턴 (Bad)**: "영어용 장식 폰트만 준비하고 다른 언어는 시스템 기본 폰트로 폴백시켜, 언어 전환 시 게임 분위기가 무너지고 일부 글리프는 누락(두부 문자)으로 표시"

**Case 4: 문화적 금기 사전 검수**
- **좋은 예 (Good)**: "종교 상징·역사적 상징·손동작이 포함된 아트 에셋을 시장별 문화 체크리스트로 사전 검수하고, 문제 소지가 있는 에셋은 지역 변형 버전을 파이프라인에서 스위칭 가능하게 설계"
- **안티패턴 (Bad)**: "문화 검수 없이 출시한 뒤 특정 시장에서 종교·문화적 논란이 발생하여 사과문 게시와 긴급 패치로 대응. 수정 비용보다 브랜드 신뢰 손실이 더 큼 (실제로 대형 타이틀들이 음악·상징물 문제로 출시 후 콘텐츠를 삭제·교체한 사례가 반복적으로 발생)"

**Case 5: 자막 vs 더빙 전략과 스트링 프리즈**
- **좋은 예 (Good)**: "1차 언어는 풀더빙, 2차 언어는 자막+영어 음성으로 계층화하여 예산 배분. 더빙 언어는 스트링 프리즈 이후 대본 변경을 금지하고, 불가피한 수정은 재녹음 비용을 명시한 변경 승인 절차를 통과해야 함"
- **안티패턴 (Bad)**: "전 언어 풀더빙을 선언했으나 프리즈 이후에도 대사 수정이 반복되어 언어별 재녹음 비용이 폭증하고, 일부 언어는 음성과 자막 내용이 불일치한 채 출시"

**Case 6: 지역별 등급·확률 공시 대응**
- **좋은 예 (Good)**: "기획 단계에서 대상 시장별 심의 등급 목표(예: GRAC 15세, PEGI 16)를 정하고 폭력·도박 표현 수위를 역산 설계. 확률형 아이템은 한국의 확률 공시 의무(2024년 시행)와 중국의 확률 공개 요건을 충족하도록 확률 테이블 노출 UI를 처음부터 설계"
- **안티패턴 (Bad)**: "출시 직전 심의에서 특정 표현이 목표 등급을 초과하여 콘텐츠 재작업으로 출시 연기. 확률 공시 요건을 뒤늦게 파악하여 지역별 빌드 분기가 급조됨"

**Case 7: 기계 번역 출시와 실행 환경 미검증 (Song of the Prairie)**
- **사건**: 어색한 기계 번역 영어에 더해 '시스템 언어를 영어로 바꿔야 실행되는' 기술 결함까지 겹쳐 글로벌 진출에 실패
- **교훈**: L10N 검증 범위는 번역문 품질만이 아니라 실행 환경(OS 로케일·인코딩·입력기)과 문화 맥락까지 포함해야 함. 기계 번역 그대로의 출시는 게임성 평가 이전에 신뢰를 무너뜨림

## 8. 핵심 인사이트 (Key Insights)

- **i18n은 설계, L10N은 실행**: 다국어를 담을 수 있는 구조(스트링 키·플레이스홀더·복수형)는 프로토타입 단계에 만들어야 합니다. 나중에 넣는 i18n은 항상 전면 재작업입니다.
- **Pseudo-loc을 조기에**: 번역이 시작되기 전에 의사 로컬라이제이션 빌드로 하드코딩·잘림·문자셋 문제를 잡으세요. 실제 번역 후 발견되는 문제는 수정 비용이 몇 배로 커집니다.
- **컨텍스트가 품질을 결정**: 번역 품질 문제의 상당수는 번역가의 실력이 아니라 컨텍스트 부족에서 나옵니다. 화자·상황·스크린샷을 스트링과 함께 전달하세요.
- **번역이 아니라 적응**: 컬처럴라이제이션은 선택이 아닙니다. 문화적 금기·상징·색상 검수를 아트·내러티브 마일스톤에 포함하여 출시 후 논란을 예방하세요.
- **규제는 역산 설계**: 심의 등급과 확률 공시는 출시 직전 체크 항목이 아니라 기획 단계의 설계 제약 조건입니다.
- **로컬라이제이션은 매출 +1/3의 레버리지 (Mike Rose, No More Robots)**: 간체 중국어·프랑스어·독일어 번역만 제대로 해도 판매량의 약 1/3이 추가된다는 인디 퍼블리셔 실측치입니다. 단 텍스트가 하드코딩되어 있으면 출시 3개월 전 착수는 불가능합니다 — 시트 내보내기→번역→재임포트 구조가 전제입니다.
- **강조 연출은 네이티브가 판단한다 (사쿠라이 마사히로)**: 글자 크기·색·애니메이션·의성어 기호 같은 강조 연출의 적정 척도는 언어권마다 다릅니다. 연출 데이터를 원 개발측이 전부 지정하지 말고 각국 번역자가 연출 스크립트를 직접 조정할 수 있게 설계하세요.
- **적정 문자 크기는 언어마다 다르다 (사쿠라이 마사히로)**: 한자는 글자당 정보량이 커서 크게+짧게, 알파벳은 작아도 읽히지만 길게 표시됩니다. 영어 전제 고정 레이아웃은 한자권에서 가독성이 파탄나므로, 휴대 모드 화면 실기 확인과 폰트 크기 선택지를 제공하세요.
