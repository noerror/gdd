# 도구·파이프라인 요구사항 작성 가이드

## 1. 목적 (Purpose)
게임 개발에 필요한 도구와 파이프라인 요구사항을 체계적으로 정의하여 개발 효율성을 극대화하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: 개발 도구, 에셋 파이프라인, 빌드 시스템, 버전 관리, 자동화
- **제외**: 구체적인 도구 구현, 서버 인프라, 배포 전략

## 3. 설계 목표 (Design Goals)

1. **효율성 (Efficiency)**: 개발 프로세스를 자동화하고 최적화합니다.
2. **일관성 (Consistency)**: 모든 팀원이 동일한 도구와 프로세스를 사용합니다.
3. **확장성 (Scalability)**: 팀 규모와 프로젝트 크기 증가에 대응합니다.
4. **안정성 (Stability)**: 도구와 파이프라인이 안정적으로 작동합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 개발 도구 (Development Tools)
- **필수 도구**: 엔진, IDE, 버전 관리
- **보조 도구**: 디버깅, 프로파일링

### 4.2 에셋 파이프라인 (Asset Pipeline)
- **에셋 제작**: 3D 모델, 텍스처, 오디오
- **에셋 처리**: 최적화, 변환, 패키징

### 4.3 빌드 & 자동화 (Build & Automation)
- **빌드 시스템**: 자동 빌드, CI/CD
- **테스트 자동화**: 유닛 테스트, 통합 테스트

### 4.4 조정 루프 & 작업 관리 (Tuning Loop & Task Management)
- **파라미터 외부화**: 조정 대상 수치를 프로그램 밖 데이터(Excel/CSV/XML)로 노출, 파라미터 간 계산식 지원(예: "이 기술은 저 기술의 1/2 위력"), 사용 빈도가 낮은 파라미터 숨김 기능(휴먼 에러 방지)
- **핫 리로드**: 수치 변경이 재빌드·재기동 없이 실행 중인 개발기에 반영되는 시간 목표(예: 10초 이내) 정의
- **디버그 모드**: 체력 임의 설정·상태 재현 등 매일 전원이 쓰는 기능 목록, 통상 플레이 컨트롤러 1개로 완결되는 조작 체계, 항목 비대화 방지를 위한 정기 정리 주기
- **티켓 기반 작업 관리**: 기획→사양서→모델→모션→프로그램→이펙트→사운드→밸런스의 흐름을 티켓으로 관리, 티켓당 리더·담당자·기한·스테이터스(진행중/완료/보류/파기) 필수 필드

### 4.5 선택 및 심화 (Optional & Advanced)
- **CI/CD 파이프라인**: 젠킨스(Jenkins)/GitHub Actions를 이용한 나잘리(Nightly) 빌드 및 자동 배포
- **연동 시스템**: 빌드 실패/성공 여부 메신저(Slack/Discord) 봇 알림 연동
- **성능 모니터링**: 에셋 임포트 시간 추적, 빌드 사이즈 증감 추이 그래프 분석

## 5. 관련 개념 (Key Concepts)

- **Asset Pipeline**: 에셋 제작부터 게임 통합까지의 프로세스
- **CI/CD**: 지속적 통합/배포
- **Version Control**: 버전 관리 시스템 (Git 등)
- **Build Automation**: 자동 빌드 시스템
- **Hot Reload**: 실행 중인 게임에 데이터 변경을 즉시 반영하는 기능. 이터레이션 속도의 핵심 지표
- **모듈러 킷 (Modular Kit)**: 표준 규격(풋프린트·피벗·문틀 치수)을 공유하는 레벨 조립 부품 세트. 규격 동결이 전제 조건
- **스토리 스크립팅 언어**: 작가가 엔지니어 없이 분기·변수·조건을 직접 작성하는 도구(Yarn Spinner, Ink 등). 내러티브 물량의 전제 조건

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 도구
- [ ] 모든 필수 도구가 정의되었는가?
- [ ] 도구 간 호환성이 확인되었는가?
- [ ] 팀 전체가 동일한 도구를 사용하는가?

### 6.2 파이프라인
- [ ] 에셋 파이프라인이 자동화되었는가?
- [ ] 에셋 처리 시간이 최적화되었는가?
- [ ] 에셋 버전 관리가 가능한가?

### 6.3 빌드 & 자동화
- [ ] 자동 빌드가 구축되었는가?
- [ ] 테스트가 자동화되었는가?
- [ ] 빌드 실패 시 알림이 제공되는가?

### 6.4 조정 루프 & 작업 관리
- [ ] 파라미터 수치 변경이 게임에 반영되기까지의 시간이 측정·목표화되었는가?
- [ ] 디버그 기능이 통상 플레이하는 컨트롤러 1개로 완결되는가?
- [ ] 각 작업의 현재 담당자(공의 소재)가 티켓으로 가시화되는가?
- [ ] 모듈러 킷의 피벗·문틀 규격이 프로젝트 초기에 확정·동결되었는가?

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 에셋 파이프라인**
- **좋은 예 (Good)**: "3D 모델 → 자동 LOD 생성 → 텍스처 압축 → 게임 엔진 임포트, 모든 과정이 자동화되어 아티스트가 버튼 클릭만으로 완료"
- **안티패턴 (Bad)**: "수동으로 모든 에셋을 처리하여 시간 낭비 및 실수 발생"

**Case 2: CI/CD**
- **좋은 예 (Good)**: "코드 커밋 → 자동 빌드 → 유닛 테스트 → 통합 테스트 → 자동 배포, 모든 과정이 자동화되어 빠른 피드백"
- **안티패턴 (Bad)**: "수동 빌드로 시간 소요 및 빌드 실패 늦게 발견"

**Case 3: 버전 관리**
- **좋은 예 (Good)**: "Git으로 모든 코드·에셋 관리, 브랜치 전략 명확, 충돌 해결 프로세스 정의"
- **안티패턴 (Bad)**: "버전 관리 없이 파일 공유로 충돌 및 데이터 손실"

**Case 4: Fortnite - 대규모 멀티플레이어 CI/CD 파이프라인**
- **게임 특성**: 전 세계 3억 이상 플레이어의 크로스플랫폼 배틀로얄 게임
- **파이프라인 전략**: 일일 수십 회 배포를 지원하는 자동화된 CI/CD 시스템
- **구체적 구현**:
  - 소스 코드 커밋 시 자동 빌드 3분 내 완료 (병렬 처리로 시간 단축)
  - 플랫폼별 자동 컴파일: PC/PS4/Xbox/Nintendo Switch/Mobile 동시 빌드 (약 45분 소요)
  - 머신러닝 기반 스모크 테스트 자동화로 450개 핵심 기능 30분 내 검증
  - A/B 테스트 환경에서 새 기능 10,000명 플레이어 대상 실시간 검증
  - Discord/Slack 봇 연동으로 빌드 결과 즉시 알림 (성공률 99.7% 유지)
  - CDN을 통한 자동 배포로 전 지역 동시 업데이트 가능 (평균 배포 시간 2시간)
- **효과**: 일주일에 평균 4회 패치 배포 가능, 버그 발견→수정 사이클 48시간 단축

**Case 5: No Man's Sky - 대규모 에셋 파이프라인 자동화**
- **게임 특성**: 절차 생성 기반 우주 탐험 게임, 수억 개 행성·생물·구조물 존재
- **파이프라인 전략**: 절차 생성 에셋의 자동 검증 및 최적화 파이프라인
- **구체적 구현**:
  - 절차 생성 3D 모델 자동 LOD 4단계 생성 (고화질/중화질/저화질/극저화질)
  - 텍스처 자동 압축: PNG 원본(평균 45MB) → ASTC 포맷 압축(2.5MB) 94.4% 용량 감소
  - 메모리 임계값 실시간 모니터링: 단일 플레이어 세션 메모리 사용량 3GB 유지
  - 성능 자동 프로파일링: GPU 렌더링 시간 16.67ms(60FPS) 이하 유지 확인
  - 빌드 시간 최적화로 증분 빌드(Incremental Build) 2분, 전체 빌드 12분 달성
- **효과**: 개발자 효율성 60% 증대, 에셋 제작→게임 통합 사이클 1주일→1일로 단축

**Case 6: Warframe - 라이브 서비스 자동화 파이프라인**
- **게임 특성**: 지속적 콘텐츠 업데이트의 PvE 협력 슈팅 게임 (12년 이상 운영 중)
- **파이프라인 전략**: 주간 업데이트를 지원하는 완전 자동화된 콘텐츠 배포 시스템
- **구체적 구현**:
  - 콘텐츠 통합 시스템: 미션 데이터(JSON) → 자동 검증 → 서버 배포 (수동 작업 0%)
  - 데이터베이스 마이그레이션 자동화: 기존 플레이어 데이터 손실 없이 주간 업데이트 적용
  - 롤백 시스템: 오류 발견 시 자동 이전 버전 복구 (평균 복구 시간 5분)
  - 플레이어 데이터 백업: 시간당 1회 자동 백업, 최근 4주 보관
  - 밸런스 시뮬레이션 파이프라인: 신규 무기 추가 시 AI 플레이어 10,000회 테스트로 윤곽(Outlier) 감지
  - 멀티플레이어 네트워크 성능 모니터링: 글로벌 플레이어 동접 자동 부하 분산 (평균 레이턴시 <50ms)
- **효과**: 주간 업데이트 안정성 99.2% 유지, 플레이어 이탈률 3% (높은 만족도 유지)

**Case 7: 파라미터 조정 파이프라인 (대난투 스매시브라더스)**
- **좋은 예 (Good)**: "전 파라미터를 Excel로 정비해 계산식 지원, 애드인 버튼 하나로 실행 중인 개발기에 변경 수치 즉시 반영, 저빈도 파라미터는 숨김 처리로 휴먼 에러 방지"
- **안티패턴 (Bad)**: "초기작 시절 컨버트 1회 10분+게임 재기동 필요, Excel 없이 텍스트로 전 파라미터 수기 관리 — 가장 많은 시행착오가 일어나는 조정 루프가 최저속 구간"

**Case 8: 스프레드시트 → 게임 데이터 리임포트 (Watch Dogs 2)**
- **좋은 예 (Good)**: "시민 AI 반응 매트릭스 수천 규칙을 Excel에서 작성해 XML로 파싱, 게임 실행 중에 시트를 수정하고 리임포트 가능해 테스트 사이클 단축. 규칙 작성 가속을 위해 엔진 내 전용 툴 대신 Excel VBA 매크로를 선택"
- **안티패턴 (Bad)**: "엔진 내 전용 에디터를 처음부터 개발하느라 규칙 작성 착수가 지연되고, 디자이너가 익숙한 표 편집 워크플로우를 버림"

**Case 9: 모듈러 킷 규격 동결 (Fallout 4)**
- **좋은 예 (Good)**: "프로젝트 초기에 싱글/더블 도어프레임 2종 치수를 전 킷에 강제해 지하실→벽돌→호텔 킷을 커스텀 아트 없이 호환, 캐릭터·애니메이션 팀에도 '최소 초크포인트' 확정 수치 제공. 명명 규칙 기반 프리팹으로 러프를 잡아 신규 건축 변형 킷 2주, 벽지 변형 2일에 산출"
- **안티패턴 (Bad)**: "피벗 포인트를 중간에 변경해 기존 배치 전체가 어긋나고, 용도가 겹치는 킷이 난립해 킷마다 설계·교육·버그 대응 오버헤드 중복"

## 8. 핵심 인사이트 (Key Insights)

- **자동화 우선**: 반복적인 작업을 자동화하여 개발 효율성을 극대화하세요.
- **일관된 도구**: 팀 전체가 동일한 도구와 프로세스를 사용하도록 표준화하세요.
- **빠른 피드백**: CI/CD로 빌드 실패를 빠르게 발견하고 대응하세요.
- **에셋 파이프라인 최적화**: 에셋 처리 시간을 최소화하여 아티스트 생산성을 높이세요.
- **"프로그램이 골격, 파라미터가 근육" (사쿠라이 마사히로)**: 조정하고 싶은 거의 모든 수치를 프로그램 밖 파라미터로 노출하는 것이 대원칙입니다. 밸런스 조정은 가장 많은 시행착오가 일어나는 곳이므로 조정 파이프라인 투자가 툴 투자의 1순위입니다.
- **스프레드시트는 AAA의 표준 파이프라인이다 (Watch Dogs 2)**: 디자이너가 엔진보다 시트에서 더 오래 일하는 프로젝트도 흔합니다. 시트 수정→게임 반영의 왕복 시간을 스펙으로 정의하고 관리하세요.
- **킷은 부채다 (Fallout 4)**: 킷마다 설계·교육·버그 대응 비용이 붙습니다. 용도 중복 킷은 통합하고, 세분화 필요는 프리팹으로 흡수하세요. 민무늬 그레이박스 킷은 비전 전달에 실패하므로 '중요한 디테일이 들어간' 러프 킷을 쓰세요.
- **스케치→러프→확증 단계를 분리하라 (Marvel's Spider-Man)**: 블록당 큐브 1개짜리 도시 스케치(1~2일)로 크기감만 검증하고, 축소 테스트 도시로 전 메트릭을 실측한 뒤에만 전진 — "도시를 두 번 지을 시간은 없다". 재작업 비용이 급증하는 지점에 확증(confirm) 게이트를 두세요.
- **작가용 스토리 언어가 물량의 전제 조건**: Yarn Spinner(Night in the Woods, 약 14,000단어 분기 대화)·Ink(80 Days)처럼 작가가 엔지니어 없이 분기 로직을 작성·퍼블리시할 수 있으면 주 2~3개 신규 퀘스트급 산출이 가능해집니다.
- **솔로 개발이라도 첫날부터 소스 컨트롤**: 로컬 PC 하나에 소스를 두면 하드웨어 고장 1회로 전체가 소실될 수 있고, 엉킨 파일 구조는 협업자 합류 시 온보딩 비용이 됩니다. 파일 아키텍처와 버전 관리는 개발 초기에 세팅하세요.
- **2주 주기 래피드 프로토타이핑 (Hitman)**: 유저 리서치 팀과 붙어 빌드를 찍으면 당일 평가, 2주 후 반복. 디지털 이전에 종이 프로토타입을 활용하세요 — "종이는 반으로 자르고 덧칠하기 쉽다".
- **프로토타입 블록의 함정 (Titanfall)**: ① 완벽주의로 블록이 비대해짐(퀵&더티 유지), ② 프로토타입은 실제 제작비를 과소평가하게 만듦, ③ 게임플레이 프로토타이핑과 레벨 흐름 설계는 별개 작업 — 추상 블록은 레벨 디자인 단계에서 실세계로 번역하세요.
