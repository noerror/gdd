# 1. GDD의 핵심 목표 및 대상

- **목표:** 게임 개발에 필요한 모든 디자인 의사결정과 구현 방향을 **명확하고 일관되게** 문서화하여, 팀 전체가 동일한 비전을 공유하고 효율적으로 개발할 수 있도록 하는 것.
- **핵심:** "무엇을 만들고, 어떻게 작동하며, 왜 재미있는가?"에 대한 구체적이고 실행 가능한 답변을 제공.
- **대상:** 개발팀 전체(디자이너, 프로그래머, 아티스트, 프로듀서), 외부 파트너, 투자자 등 프로젝트 이해관계자 모두.
- **재구현 가능성:** 이 문서만으로 같은 시스템을 다시 구현할 수 있을 정도로 구체적이어야 한다.

# 2. GDD 작성 핵심 원칙

## 2-1. 구조와 관리 (Structure & Maintenance)

1. **단일 진실 소스(SSoT):** 각 문서는 해당 영역의 유일한 기준점이며, 정보 중복을 피하고 상호 참조(Cross Reference)로 연결하여 유지보수성을 확보합니다.
2. **구조의 평탄화 (3단계 이내):** 과도한 그룹화로 깊이를 늘리기보다, 항목을 적절히 나열하여(Flatten) 정보 도달 깊이를 3단계 이내로 유지합니다. (예: 깊고 좁은 구조 → 얕고 넓은 구조)
3. **계층적 설계:** 문서는 개념(Concept) → 메커닉(Mechanic) → 규칙(Rule) → 파라미터(Parameter) → 예외/경계(Edge) 순서로 위에서 아래로 좁혀가며 구체화합니다.
4. **규칙과 파라미터 분리:** 어떤 일이 일어나는지(규칙)와 얼마나 일어나는지(수치)를 별도 섹션으로 분리하여 밸런싱 변경에 유연하게 대응합니다.
5. **유지보수와 확장성:** 변경이 쉽고, 수정 사항이 다른 부분에 미치는 영향을 추적할 수 있어야 합니다.
6. **지속적 개선:** 완벽을 기다리기보다는 70~80% 수준으로 일단 공유하고 피드백을 받아 수정하며, 불필요한 내용은 과감히 삭제(킬 더 달링)합니다.

## 2-2. 명확성과 완전성 (Clarity & Completeness)

1. **명확성과 단순함:** 누가 읽어도 동일하게 이해되도록 직관적으로 작성하며, 정보 구조는 계층적으로 작성해서 너무 많은 내용을 포함하지 않도록 합니다.
2. **완전한 일관성:** 문서 간, 섹션 간 모순이 없어야 하며, 용어, 포맷, 스타일을 전체 프로젝트에서 통일합니다.
3. **완전성(Completeness):** 정상적인 흐름뿐만 아니라 모든 예외 케이스와 엣지 케이스를 고려하여 누락 없는 정보를 제공합니다.
4. **시스템 목적 명시:** 각 시스템이 해결하려는 문제와 도입 목적을 처음에 분명히 드러냅니다.
5. **발동 조건과 효과 명확화:** 언제 발동되는지(조건/트리거), 어떻게 처리되는지(흐름), 무엇이 결과로 나오는지(효과)를 명확히 정의합니다.
6. **상태와 데이터 구조 정의:** 시스템이 관리하는 상태값과 데이터 구조를 빠짐없이 정의합니다.

## 2-3. 실행과 검증 (Execution & Verification)

1. **구체적인 실행 가능성:** 추상적인 개념보다는 프로그래머가 직접 구현 가능한 수준의 구체적인 규칙, 데이터, 공식을 포함합니다.
2. **측정 가능성:** 성공/실패 기준과 목표를 구체적인 수치로 표현하여 검증 가능하게 합니다.
3. **핵심 흐름 정의:** 조건-판정-효과가 단계별로 이해될 수 있도록 처리 흐름을 구체화합니다.
4. **예외 및 경계 처리:** 극단값(최대/최소), 동시 요청, 실패 상황, 잘못된 입력 등의 처리 방침을 명시합니다.
5. **시스템 간 연동 명시:** 의존하는 시스템, 이벤트 발행/구독, 변경 시 영향받는 범위를 정리합니다.
6. **튜닝 가능 파라미터 분리:** 나중에 밸런스 조정할 항목을 한곳에 모아 조정 가능 범위까지 표시합니다.

## 2-4. 기획 철학 (Design Philosophy)

1. **개념은 한 줄로 고정:** 모든 디테일은 바뀔 수 있지만, 시스템의 핵심 개념은 한 줄 문장으로 고정합니다.
2. **경험 우선, 알고리즘 후순위:** 플레이어가 어떻게 느끼느냐가 1순위, 계산 방식은 그걸 구현하는 수단입니다.
3. **예측 가능하되 반복 회피:** 플레이어가 대략 결과를 예상할 수 있되, 완전히 똑같이 반복되지는 않게 설계합니다.
4. **악용까지 고려:** 의도한 사용 시나리오뿐만 아니라 예상되는 악용/극단 시나리오까지 포함해 경계를 정의합니다.
5. **패턴으로 사고:** 개별 시스템이라도 다른 곳에서 재사용 가능한 패턴 수준으로 생각합니다.

# 3. GDD 작성 실전 팁

## 3-1. 구조화 원칙

1. **피라미드 구조:** 결론(요약)으로 시작해 핵심 규칙→세부 규칙→예외→예시 순으로 전개한다.
2. **항목화 우선, 서술 최소화:** 기본은 짧은 문단과 불릿 포인트로 작성한다. 단, 파라미터/확률 테이블/상태 전이처럼 수치나 관계가 복잡한 경우에는 표나 다이어그램을 사용한다.
3. **예시는 조건부로만 사용:** 규칙만으로 해석 차이가 날 여지가 있거나, 상태·순서가 복잡해 오해가 예상될 때, 과거 해석 충돌 전례가 있을 때, QA 핵심 테스트 케이스일 때는 예시를 [상태]→[행동]→[결과] 형식으로 1~2줄로만 추가한다. 규칙을 반복 설명하지 않는다.
4. **즉시 실행 가능:** 모든 규칙은 발동 조건/처리 흐름/효과/예외/성공 기준을 항목으로 분리해 한눈에 보이게 적는다.
5. **간결 유지:** 한 항목은 3~5줄 이내로 유지하고, 불필요한 수식어를 제거한다.
6. **맥락 노트:** 논쟁 소지가 있는 결정에만 [메모] 형식으로 1~2줄의 근거를 남겨 맥락을 보존한다.
7. **단일 출처:** 동일 정보는 한 문서에만 두고, 나머지는 그 문서를 링크해 중복을 막는다. 특히 파라미터, 공통 용어, 보상 테이블은 단일 출처를 강제한다.
8. **문제 흐름:** "경험 목표 한 줄 → 현재 문제 → 해결" 순서로 서술해 맥락을 명확히 한다.

## 3-2. 계층별 필수 구성

각 시스템 문서는 다음 계층을 따라 작성합니다:

1. **개요 레벨 (Concept):** 시스템 한 줄 설명, 도입 목적/문제, 상위 시스템 역할
2. **메커닉 레벨 (Mechanic):** 구성 메커닉 A, B, C... 개요
3. **규칙 레벨 (Rule):** 각 메커닉의 발동 조건, 처리 흐름, 효과
4. **데이터 레벨 (Parameter):** 파라미터 표, 상태/데이터 구조, 튜닝 범위
5. **예외 레벨 (Edge):** 극단값, 실패·중단 처리, 시스템 간 충돌/연동

## 3-3. 언어 사용

1. **능동태 사용:** "플레이어는 주사위를 굴린다" (O) vs "주사위가 굴려진다" (X)
2. **명확한 용어:** 동일한 개념에 여러 용어를 섞어 쓰지 않습니다.
3. **수치로 표현:** "빠르게"(X) → "0.2초 이내"(O)
4. **조건 명시:** "대부분"(X) → "80% 이상"(O)
5. **모호한 표현 금지:** "적절히", "알아서", "대충" 같은 표현을 사용하지 않습니다.

## 3-4. 검증 체크리스트

## 3-4. 검증 체크리스트

### 문서 시작 부분 (첫 5줄)
- [ ] "이 시스템은 [무엇]을 통해 [어떤 문제]를 해결한다" 형식의 문장이 있는가?
- [ ] 상위 시스템/다른 문서로의 링크가 명시되어 있는가?

### 메커닉/규칙 섹션
- [ ] 모든 메커닉에 "발동 조건 / 처리 / 효과" 3가지가 있는가?
- [ ] "적절히", "알아서", "충분히" 같은 모호한 표현이 없는가?
- [ ] 수치가 필요한 곳에 "빠르게" 대신 "0.2초"처럼 구체적 값이 있는가?
- [ ] 조건에 "대부분", "보통" 대신 "80% 이상", "평균" 같은 명확한 기준이 있는가?

### 예외 처리 (가장 자주 누락)
- [ ] 최대값 입력 시 동작이 명시되어 있는가?
- [ ] 최소값 또는 0 입력 시 동작이 명시되어 있는가?
- [ ] 인벤토리/슬롯 풀(full) 상황 처리가 있는가?
- [ ] 네트워크 끊김/지연 시 처리가 있는가?
- [ ] 중간 이탈/강제 종료 시 데이터 처리가 있는가?
- [ ] 동시 요청/Race Condition 처리가 있는가?

### 파라미터 섹션
- [ ] 파라미터가 규칙 설명과 분리된 별도 섹션에 있는가?
- [ ] 각 파라미터에 기본값이 명시되어 있는가?
- [ ] 조정 가능한 범위(최소~최대)가 표시되어 있는가?
- [ ] 수치 단위(초, m/s, %, 개)가 명확한가?

### 연동성
- [ ] 이 시스템이 읽는(GET) 다른 시스템 데이터가 명시되어 있는가?
- [ ] 이 시스템이 쓰는(SET) 다른 시스템 데이터가 명시되어 있는가?
- [ ] 이 시스템이 발행하는 이벤트가 나열되어 있는가?
- [ ] 이 파라미터를 바꾸면 영향받는 다른 시스템이 명시되어 있는가?

### 용어 일관성
- [ ] 같은 개념을 여러 이름으로 부르는 곳이 없는가?
- [ ] 용어집/상위 문서로 정의 링크가 있는가?
- [ ] 약어를 처음 쓸 때 풀네임을 적었는가?

### 중복 제거
- [ ] 이 정보가 다른 문서에도 있다면, 한쪽을 링크로 바꿨는가?
- [ ] 파라미터를 두 군데 이상에서 정의하고 있지 않은가?

### 구현 가능성 테스트
- [ ] 개발자에게 이 문서를 주고 "이것만으로 구현하세요"라고 말할 수 있는가?
- [ ] QA에게 이 문서에서 테스트 케이스를 뽑으라고 할 수 있는가?
- [ ] 6개월 후 다시 읽어도 왜 이렇게 정했는지 이해할 수 있는가?

## 3-5. 마크다운 작성 가이드

1. **파라미터는 간결하게:**
  - 기본: `파라미터명: 값 (설명)` 형태로 한 줄 작성
  - 예: `이동속도: 5.0 m/s (범위 3.0~8.0)`
2. **표는 최소화:**
  - 조건: 파라미터가 5개 이상이고 속성이 3개 이상일 때만 사용
  - 예: 레벨별 스탯 테이블 (레벨 1~20, 각 레벨마다 HP/공격/방어/크리티컬/속도)
3. **상태 전이는 화살표:**
   - `대기 → 이동 (입력: 방향키)`
   - `이동 → 공격 (조건: 적 발견 AND 거리 < 5m)`
4. **수식은 인라인 코드:**
   - 최종 데미지 = `기본 데미지 × (1 + 공격력 / 100)`
5. **복잡한 관계는 텍스트 설명 + 예시:**
   - 설명: "레벨이 높을수록 경험치 요구량이 기하급수적으로 증가"
   - 예시: `Lv1→2: 100exp, Lv2→3: 150exp, Lv3→4: 225exp`
