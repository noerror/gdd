# QA 계획 & 테스트 케이스 작성 가이드

## 1. 목적 (Purpose)
게임의 QA 계획과 테스트 케이스를 체계적으로 작성하여 출시 전 품질을 보장하고 버그를 최소화하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: QA 전략, 테스트 케이스, 테스트 주기, 버그 트래킹, 자동화 테스트
- **제외**: 구체적인 테스트 도구 구현, 성능 테스트 세부사항

## 3. 설계 목표 (Design Goals)

1. **높은 품질 (High Quality)**: 출시 전 모든 주요 버그를 발견하고 수정합니다.
2. **효율적인 테스트 (Efficient Testing)**: 제한된 시간과 리소스로 최대 커버리지를 달성합니다.
3. **자동화 (Automation)**: 반복적인 테스트를 자동화하여 효율성을 높입니다.
4. **명확한 기준 (Clear Criteria)**: 출시 기준이 명확하고 측정 가능합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 QA 전략 (QA Strategy)
- **테스트 범위**: 기능·성능·호환성
- **테스트 주기**: 일일·주간·마일스톤

### 4.2 테스트 케이스 (Test Cases)
- **기능 테스트**: 모든 기능 검증
- **회귀 테스트**: 수정 후 재검증
- **사용자 시나리오**: 실제 플레이 경로

### 4.3 버그 관리 (Bug Management)
- **버그 우선순위**: Critical·High·Medium·Low
- **버그 트래킹**: 발견·수정·검증

### 4.4 선택 및 심화 (Optional & Advanced)
- **자동화 테스트**: 빌드 배포 시 자동 스모크 테스트(Smoke Test) 스크립트, 주요 API 응답 속도 모니터링
- **호환성 테스트**: OS 버전별(Windows/Android/iOS) 호환성 매트릭스, 최소/권장 사양 기기 리스트
- **데이터 검증**: 대규모 더미(Dummy) 데이터 생성 후 DB 로드 테스트, 밸런스 시뮬레이션 결과 검증

## 5. 관련 개념 (Key Concepts)

- **Test Case**: 특정 기능을 검증하는 테스트 시나리오
- **Regression Testing**: 수정 후 기존 기능 재검증
- **Bug Priority**: 버그의 심각도와 우선순위
- **Test Automation**: 자동화된 테스트

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 QA 전략
- [ ] 테스트 범위가 명확하게 정의되었는가?
- [ ] 테스트 주기가 개발 일정과 동기화되었는가?
- [ ] QA 팀의 역할이 명확한가?

### 6.2 테스트 케이스
- [ ] 모든 주요 기능에 테스트 케이스가 있는가?
- [ ] 테스트 케이스가 명확하고 재현 가능한가?
- [ ] 엣지 케이스가 고려되었는가?

### 6.3 버그 관리
- [ ] 버그 우선순위 기준이 명확한가?
- [ ] 버그 트래킹 시스템이 구축되었는가?
- [ ] Critical 버그의 수정 기한이 정의되었는가?

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 테스트 케이스**
- **좋은 예 (Good)**: "TC001: 플레이어 점프 - 전제조건: 평지에 서 있음 / 단계: 점프 버튼 누름 / 예상 결과: 캐릭터가 1m 높이로 점프, 명확하고 재현 가능"
- **안티패턴 (Bad)**: "점프 테스트 - 점프가 작동하는지 확인" (모호함)

**Case 2: 버그 우선순위**
- **좋은 예 (Good)**: "Critical: 게임 크래시 (24시간 내 수정) / High: 주요 기능 불가 (3일 내) / Medium: 밸런스 문제 (1주 내) / Low: UI 오타 (출시 전), 명확한 기준"
- **안티패턴 (Bad)**: "모든 버그를 동일하게 취급하여 Critical 버그 방치"

**Case 3: 자동화 테스트**
- **좋은 예 (Good)**: "유닛 테스트: 핵심 로직 검증 / 통합 테스트: 시스템 간 상호작용 / 회귀 테스트: 매일 자동 실행, 효율적인 테스트"
- **안티패턴 (Bad)**: "모든 테스트를 수동으로 진행하여 시간 낭비"

**Case 4: The Witcher 3 - 대규모 오픈월드 회귀 테스트**
- **게임 특성**: 100시간+ 콘텐츠의 대규모 오픈월드 RPG, 복잡한 시스템 상호작용
- **테스트 전략**: 모듈별 자동화 테스트와 수동 시나리오 테스트의 혼합
- **구체적 구현**:
  - 자동화 테스트: 130개 핵심 퀘스트 진행 경로 자동 검증 (매일 실행, 약 45분 소요)
  - 회귀 테스트 케이스:
    - TC001: 메인 퀘스트 'White Orchard' 100% 완료 후 다음 지역 진입 확인
    - TC002: 삼각 관계 플래그(Flag) 변수 상태 추적: A사랑 선택 시 B 이벤트 스킵 검증
    - TC003: 플레이어 인벤토리 999개 아이템 보유 상태에서 신규 아이템 습득 시 드롭 로직 정상 작동
  - 성능 회귀 테스트: 노비그래드 지역 내 200개 NPC 동시 렌더링 시 FPS 유지도 확인 (목표: 60FPS 유지)
  - 크래시 발생 시나리오 테스트: 2,000시간의 랜덤 플레이 자동 시뮬레이션으로 엣지 케이스 발견
- **효과**: 출시 후 Critical 버그 80% 감소, 플레이어 커뮤니티 평가 9.3/10

**Case 5: Red Dead Redemption 2 - 극단적 상황 테스트 케이스**
- **게임 특성**: 초대규모 오픈월드, 300개 이상의 상호작용 가능한 객체
- **테스트 전략**: 극단적 상황(Extreme Scenario)과 엣지 케이스 우선 테스트
- **구체적 구현**:
  - 극단적 상황 테스트:
    - TC101: 폭우(100mm/시간) + 낙뢰 + 안개 + 홍수 동시 발생 상황에서 캐릭터 이동 가능성 검증
    - TC102: NPC 100명 동시 총격전 상황에서 프레임드롭 최대치 측정 (목표: 30FPS 이상 유지)
    - TC103: 말과 마차 충돌 시 물리 엔진 오류(Clipping) 발생 여부 확인
  - 호환성 테스트 매트릭스:
    - Windows 10/11, PS4/Pro, Xbox One/Series X/S 6가지 플랫폼
    - 각 플랫폼별 메모리 용량 테스트 (최소 8GB ~ 최대 16GB)
    - 외장 드라이브 설치 시 로딩 시간 측정 (목표: 60초 이내)
  - 데이터 무결성 테스트: 200시간 플레이 후 저장 파일 손상 여부 검증
- **효과**: 플레이 중 CTD(Crash To Desktop) 발생률 0.03%, 플레이어 이탈률 5% (매우 낮음)

**Case 6: Final Fantasy XIV - 라이브 서비스 지속 테스트**
- **게임 특성**: 지속적 업데이트(패치)의 MMORPG, 운영 중 안정성 필수
- **테스트 전략**: 주간 패치 배포 전 자동화와 수동 테스트의 병렬 실행
- **구체적 구현**:
  - 스모크 테스트 (패치 배포 2시간 전):
    - TC201: 게임 로그인 및 캐릭터 로드 시간 <5초 확인
    - TC202: 주요 던전 3곳 입장 및 몬스터 전투 진행 가능성 검증
    - TC203: 마켓보드(경매장) 거래 기능 정상 작동 확인
  - 회귀 테스트 (자동화):
    - 550개 NPC 퀘스트 로직 재검증
    - 신규 아이템 드롭 확률 시뮬레이션 (1,000회 반복 테스트)
    - 길드 은행 시스템 동시 접근 안정성 (500명 동시 접근)
  - 성능 테스트:
    - 대규모 전투 이벤트(레이드): 64명 동시 플레이 시 네트워크 레이턴시 <100ms 유지
    - 서버 부하 테스트: 일일 피크타임(저녁 8-10시) 동시 접속자 수 시뮬레이션
  - 데이터 검증: 매 패치 후 플레이어 계정 데이터 백업 검증, 롤백 테스트 월 1회 실행
- **효과**: 12년 연속 운영 중 치명적 서버 오류 0건, 플레이어 만족도 8.8/10

## 8. 핵심 인사이트 (Key Insights)

- **테스트 자동화**: 반복적인 테스트를 자동화하여 QA 효율성을 극대화하세요.
- **명확한 우선순위**: Critical 버그는 24시간 내 수정하는 등 명확한 기준을 수립하세요.
- **조기 테스트**: 개발 초기부터 테스트를 시작하여 버그를 조기에 발견하세요.
- **실제 사용자 시나리오**: 실제 플레이어가 경험할 시나리오를 테스트하여 실용적인 품질을 보장하세요.
