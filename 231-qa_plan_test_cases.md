# QA 계획 & 테스트 케이스 작성 가이드

## 1. 목적 (Purpose)
게임의 QA 계획과 테스트 케이스를 체계적으로 작성하여 출시 전 품질을 보장하고 버그를 최소화하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: QA 전략, 테스트 케이스, 테스트 주기, 버그 트래킹, 자동화 테스트
- **제외**: 구체적인 테스트 도구 구현, 성능 테스트 세부사항

## 3. 설계 목표 (Design Goals)

1. **높은 품질 (High Quality)**: 출시 전 모든 주요 버그를 발견하고 수정합니다.
2. **효율적인 테스트 (Efficient Testing)**: 제한된 시간과 리소스로 최대 커버리지를 달성합니다.
3. **자동화 (Automation)**: 반복적인 테스트를 자동화하여 효율성을 높입니다.
4. **명확한 기준 (Clear Criteria)**: 출시 기준이 명확하고 측정 가능합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 QA 전략 (QA Strategy)
- **테스트 범위**: 기능·성능·호환성
- **테스트 주기**: 일일·주간·마일스톤

### 4.2 테스트 케이스 (Test Cases)
- **기능 테스트**: 모든 기능 검증
- **회귀 테스트**: 수정 후 재검증
- **사용자 시나리오**: 실제 플레이 경로

### 4.3 버그 관리 (Bug Management)
- **버그 우선순위**: Critical·High·Medium·Low
- **버그 트래킹**: 발견·수정·검증

### 4.4 선택 및 심화 (Optional & Advanced)
- **자동화 테스트**: 빌드 배포 시 자동 스모크 테스트(Smoke Test) 스크립트, 주요 API 응답 속도 모니터링
- **호환성 테스트**: OS 버전별(Windows/Android/iOS) 호환성 매트릭스, 최소/권장 사양 기기 리스트
- **데이터 검증**: 대규모 더미(Dummy) 데이터 생성 후 DB 로드 테스트, 밸런스 시뮬레이션 결과 검증

## 5. 관련 개념 (Key Concepts)

- **Test Case**: 특정 기능을 검증하는 테스트 시나리오
- **Regression Testing**: 수정 후 기존 기능 재검증
- **Bug Priority**: 버그의 심각도와 우선순위
- **Test Automation**: 자동화된 테스트
- **클리넥스 테스터 (Kleenex Tester)**: 게임을 처음 보는 1회용 테스터. 재사용하면 신규 유저 관점이 오염됨 (Uncharted 3는 마지막 6개월간 21회 신규 모집)
- **락아웃 (Lockout)**: must-fix 버그 소진 후 해당 코드 영역을 동결해 연쇄(knock-on) 버그 유입을 막는 절차
- **프레임타임 일관성 (Frametime Consistency)**: 평균 FPS가 높아도 프레임 간격이 불규칙하면 체감 랙이 발생 — FPS 수치와 별도의 검증 항목

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 QA 전략
- [ ] 테스트 범위가 명확하게 정의되었는가?
- [ ] 테스트 주기가 개발 일정과 동기화되었는가?
- [ ] QA 팀의 역할이 명확한가?

### 6.2 테스트 케이스
- [ ] 모든 주요 기능에 테스트 케이스가 있는가?
- [ ] 테스트 케이스가 명확하고 재현 가능한가?
- [ ] 엣지 케이스가 고려되었는가?

### 6.3 버그 관리
- [ ] 버그 우선순위 기준이 명확한가?
- [ ] 버그 트래킹 시스템이 구축되었는가?
- [ ] Critical 버그의 수정 기한이 정의되었는가?

### 6.4 실사용 시나리오 (스트리머 첫 1시간 기준)
- [ ] 첫 실행 시 셰이더 사전 컴파일과 진행 표시가 있는가?
- [ ] 평균 FPS 외에 프레임타임 일관성·입력 지연·부분 프레임 불일치(UI/컷신/게임플레이 간)가 테스트 매트릭스에 있는가?
- [ ] 크래시 후 복귀 시 진행 손실이 30초 이내인가?
- [ ] 세이브 정책(오토세이브 유무·주기)이 첫 사망 전에 안내되는가?
- [ ] 상점 반복 거래·루팅 반복 등 경제 익스플로잇 시나리오를 실행했는가?
- [ ] 설정 변경 → 적용 → 재시작 전 조합이 테스트되었는가? (설정 소실·크래시 검출)
- [ ] 플랫폼 인증(certification) 가이드라인 정독·대응 시간이 일정에 별도 배정되었는가? (예: Switch의 '플레이 중 조이콘 분리' 시나리오)

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 테스트 케이스**
- **좋은 예 (Good)**: "TC001: 플레이어 점프 - 전제조건: 평지에 서 있음 / 단계: 점프 버튼 누름 / 예상 결과: 캐릭터가 1m 높이로 점프, 명확하고 재현 가능"
- **안티패턴 (Bad)**: "점프 테스트 - 점프가 작동하는지 확인" (모호함)

**Case 2: 버그 우선순위**
- **좋은 예 (Good)**: "Critical: 게임 크래시 (24시간 내 수정) / High: 주요 기능 불가 (3일 내) / Medium: 밸런스 문제 (1주 내) / Low: UI 오타 (출시 전), 명확한 기준"
- **안티패턴 (Bad)**: "모든 버그를 동일하게 취급하여 Critical 버그 방치"

**Case 3: 자동화 테스트**
- **좋은 예 (Good)**: "유닛 테스트: 핵심 로직 검증 / 통합 테스트: 시스템 간 상호작용 / 회귀 테스트: 매일 자동 실행, 효율적인 테스트"
- **안티패턴 (Bad)**: "모든 테스트를 수동으로 진행하여 시간 낭비"

**Case 4: The Witcher 3 - 대규모 오픈월드 회귀 테스트**
- **게임 특성**: 100시간+ 콘텐츠의 대규모 오픈월드 RPG, 복잡한 시스템 상호작용
- **테스트 전략**: 모듈별 자동화 테스트와 수동 시나리오 테스트의 혼합
- **구체적 구현**:
  - 자동화 테스트: 130개 핵심 퀘스트 진행 경로 자동 검증 (매일 실행, 약 45분 소요)
  - 회귀 테스트 케이스:
    - TC001: 메인 퀘스트 'White Orchard' 100% 완료 후 다음 지역 진입 확인
    - TC002: 삼각 관계 플래그(Flag) 변수 상태 추적: A사랑 선택 시 B 이벤트 스킵 검증
    - TC003: 플레이어 인벤토리 999개 아이템 보유 상태에서 신규 아이템 습득 시 드롭 로직 정상 작동
  - 성능 회귀 테스트: 노비그래드 지역 내 200개 NPC 동시 렌더링 시 FPS 유지도 확인 (목표: 60FPS 유지)
  - 크래시 발생 시나리오 테스트: 2,000시간의 랜덤 플레이 자동 시뮬레이션으로 엣지 케이스 발견
- **효과**: 출시 후 Critical 버그 80% 감소, 플레이어 커뮤니티 평가 9.3/10

**Case 5: Red Dead Redemption 2 - 극단적 상황 테스트 케이스**
- **게임 특성**: 초대규모 오픈월드, 300개 이상의 상호작용 가능한 객체
- **테스트 전략**: 극단적 상황(Extreme Scenario)과 엣지 케이스 우선 테스트
- **구체적 구현**:
  - 극단적 상황 테스트:
    - TC101: 폭우(100mm/시간) + 낙뢰 + 안개 + 홍수 동시 발생 상황에서 캐릭터 이동 가능성 검증
    - TC102: NPC 100명 동시 총격전 상황에서 프레임드롭 최대치 측정 (목표: 30FPS 이상 유지)
    - TC103: 말과 마차 충돌 시 물리 엔진 오류(Clipping) 발생 여부 확인
  - 호환성 테스트 매트릭스:
    - Windows 10/11, PS4/Pro, Xbox One/Series X/S 6가지 플랫폼
    - 각 플랫폼별 메모리 용량 테스트 (최소 8GB ~ 최대 16GB)
    - 외장 드라이브 설치 시 로딩 시간 측정 (목표: 60초 이내)
  - 데이터 무결성 테스트: 200시간 플레이 후 저장 파일 손상 여부 검증
- **효과**: 플레이 중 CTD(Crash To Desktop) 발생률 0.03%, 플레이어 이탈률 5% (매우 낮음)

**Case 6: Final Fantasy XIV - 라이브 서비스 지속 테스트**
- **게임 특성**: 지속적 업데이트(패치)의 MMORPG, 운영 중 안정성 필수
- **테스트 전략**: 주간 패치 배포 전 자동화와 수동 테스트의 병렬 실행
- **구체적 구현**:
  - 스모크 테스트 (패치 배포 2시간 전):
    - TC201: 게임 로그인 및 캐릭터 로드 시간 <5초 확인
    - TC202: 주요 던전 3곳 입장 및 몬스터 전투 진행 가능성 검증
    - TC203: 마켓보드(경매장) 거래 기능 정상 작동 확인
  - 회귀 테스트 (자동화):
    - 550개 NPC 퀘스트 로직 재검증
    - 신규 아이템 드롭 확률 시뮬레이션 (1,000회 반복 테스트)
    - 길드 은행 시스템 동시 접근 안정성 (500명 동시 접근)
  - 성능 테스트:
    - 대규모 전투 이벤트(레이드): 64명 동시 플레이 시 네트워크 레이턴시 <100ms 유지
    - 서버 부하 테스트: 일일 피크타임(저녁 8-10시) 동시 접속자 수 시뮬레이션
  - 데이터 검증: 매 패치 후 플레이어 계정 데이터 백업 검증, 롤백 테스트 월 1회 실행
- **효과**: 12년 연속 운영 중 치명적 서버 오류 0건, 플레이어 만족도 8.8/10

**Case 7: 스트리머 첫 1시간 시나리오 (Lirik 방송 실측)**
- **배경**: 신작 첫 방송 1시간 내 발견된 결함이 시청자 수만 명에게 실시간 노출 — 스트리머의 첫 세션이 최대 규모의 공개 QA가 됨
- **실측 결함 목록**: The Callisto Protocol 셰이더 실시간 컴파일 스터터(유저가 레딧에서 원인·우회법 검색), The Day Before 물병 되팔이 경제 익스플로잇·NPC 증발(첫 세션에 연쇄 발견), Wo Long 데모 60fps 캡+상시 스터터("데모는 최종 품질의 샘플이어야 한다"), Starfield 조준 레티클만 30fps 갱신
- **파생 원칙**: 성능 평판은 체감 전에 여론으로 먼저 도착 — Elden Ring은 실행 전에 이미 "PC 성능 문제 있다더라"가 낙인으로 작동. 출시 직전 성능 이슈는 별도 트래킹 대상
- **권장 테스트**: 라이브 전 '스트리머 1시간 시나리오'(첫 실행 → 설정 전체 변경 → 첫 사망 → 상점 왕복 거래 → 크래시 복구)를 별도 테스트 케이스로 운영

**Case 8: God of War - 2종 테스트 체계와 피드백 필터링**
- **구조**: 비유도(unguided) 플레이스루로 통증 지점 발견 → 1:1 사용성 테스트로 원인 격리, 회당 20명(통계가 아니라 플레이어 유형 스펙트럼 확보용)
- **원인 격리 사례**: "전투가 재미없다"는 피드백의 실제 원인이 카메라 감도로 판명 — 증상과 원인을 분리 수집
- **운영 원칙**: 크리에이티브 디렉터 직속 전담 팀 + 확정 캘린더 절대 취소 금지(취소 반복 = 프로세스 신뢰 붕괴), 테스트는 전사 스트리밍 + 매일 밤 핵심 발견 리포트 공유
- **안티패턴 (Bad)**: "테스터가 제안한 해결책을 그대로 구현 — Hidden Folks는 테스터 제안(손 모양 가이드)을 구현하자 '게임이 나를 아기 취급한다'는 반응. 진짜 문제는 잘못된 입력에 게임이 무반응인 것이었다"

**Case 9: 스매시브라더스 - 조합 폭발 대응 버그 파이프라인**
- **티켓 흐름**: 재현 → 티켓 → 랭크(S: 행업·세이브 파괴 / A: 진행 지장 / 하위: 보류 가능) → 섹션·담당자 배정(그래픽 깨짐은 아티스트, 모션 파탄은 모션 디자이너) → 복수 조합 재확인 → '완료 보존'(종료 티켓도 재발 대비 보관)
- **조합 폭발 대응**: 파이터 89종만으로 약 8,000개 대전 조합 — 컴퓨터 대전을 밤새 자동 실행하고, 종반에는 일본 → 유럽 → 미국으로 같은 빌드를 바통터치하며 24시간 디버그
- **버그 다발 지점 조준**: 타자를 강제 제어하는 거동(잡기·던지기), 씬 전환이 있는 기술, 무리한 지형, 파라미터·육성 요소 — 다발 지점을 알고 우선 배치
- **수정 리스크**: 수정 하나가 새 버그를 낳으므로(엔백) 수정 후 대량 재확인을 전제로 일정 산정

## 8. 핵심 인사이트 (Key Insights)

- **테스트 자동화**: 반복적인 테스트를 자동화하여 QA 효율성을 극대화하세요.
- **명확한 우선순위**: Critical 버그는 24시간 내 수정하는 등 명확한 기준을 수립하세요.
- **조기 테스트**: 개발 초기부터 테스트를 시작하여 버그를 조기에 발견하세요.
- **실제 사용자 시나리오**: 실제 플레이어가 경험할 시나리오를 테스트하여 실용적인 품질을 보장하세요.
- **아무것도 설명하지 마라**: 사전 소개는 출시 후 유저가 받지 못할 특혜입니다. 허용되는 한마디는 "아직 미완성이에요"뿐입니다. 게이머만으로 테스트하지 마세요 — 게이머는 거의 항상 더 참을성이 없고, 개발자 모임은 '아직 문제가 아닌 문제'를 지적합니다.
- **해결책이 아니라 문제 서술만 수집**: 테스터의 처방("파워업이 부족해요")이 아니라 그 밑의 증상("총이 약하게 느껴진다")을 파세요. 유저가 화났다는 사실은 데이터지만, 유저가 말하는 원인은 검증 대상입니다.
- **AI 봇으로 조합을 커버**: The Division은 1,000기의 AI 봇으로 월드·미션을 자동 테스트했고, Sea of Thieves는 AI 행동 트리의 상태 전환 자체를 회귀 테스트로 검증합니다(20분 주기 소규모 테스트 + 야간 대형 시뮬레이션, 레드 스크린이면 커밋 금지). 사람 QA는 버그 사냥 대신 경험 평가에 집중시키세요.
- **인센티브 파괴 테스트**: Diablo는 최초로 게임을 깨는 직원에게 $100 현상금을 걸어 빈사→교환→즉살 콤보를 발견하고 해당 스킬을 즉시 삭제했습니다. 파괴 동기를 공식 제도로 만드세요.
- **관찰이 의견보다 정확하다**: 사쿠라이 마사히로의 초등학생 모니터 — 의견을 '듣지' 말고 반응이 좋았던 곳·조작이 막힌 곳·질려가는 조짐의 '공기'를 관찰하세요. 길들여진 어른 테스터에게서는 나오지 않는 신호입니다.
- **베타 표본 편향을 인지하라**: FTL의 십 언락 퍼즐은 유료 베타($25를 낸 하드코어 3,000명)에서는 재미로 평가됐다가 출시 후 최다 불만 요소가 됐습니다. 같은 기능이 커뮤니티 규모·성향에 따라 정반대로 작동합니다.
- **판타지와 기능을 분리 판정**: "보트가 지루하다"(서사 미구현 = fantasy)와 "조작이 어렵다"(지금 고칠 수 있는 function)를 구분하세요. God of War는 미구현 상태의 판타지 피드백에 버텼고, 서사가 들어가자 의도가 검증됐습니다.
- **작은 결함은 테스트 도중이라도 즉시 수정**: '빌드 동결' 통념과 반대로, 5초짜리 결함(설정창이 기본으로 열려 있는 수준)이 전체 테스트 데이터를 오염시키는 것보다 그날 고치는 편이 낫습니다. 문제 발견에는 큰 표본이 필요 없습니다 — "Fail as fast as you can".
