# GDD 워크 플로우 / 프레임워크

- LLM을 활용하는 MVP 지향 개발 워크플로우 제안
- [텍스트 기반 기획서와 게임 데이터 통합](001-Text-Data-Integration.md)

# 개요

## 1. 개발 키 컨셉

- **MVP** (Minimum Viable Product, 최소기능) 개발론 지향 : 최소한의 기능을 개발하고 피드백을 받아 개선, 고도화하는 방법
- 프리프로덕션에서 **수직 슬라이스 프로토타이핑** : 웹 기반으로 LLM 기반 프로토타이핑
- **기획서와 실 동작의 일치** : LLM 및 버전 관리 / 프로그래밍 연동 / QA 최적화를 위해 문서에 데이터를 포함

## 2. 역할 정리

1. **게임 디렉터** 아이디어 발굴
2. **크리에이티브 디렉터** 아이디어 기반으로 컨셉 기획 및 기능 사양서 작성
3. 기능검증 프로토타입으로 컨셉 기획 보강 : LLM을 이용한 비 프로그래머
4. **게임 디자이너** 컨셉기획서 기반으로 GDD 작성 (핵심 메커니즘 우선 상세화)
5. **개발팀 & 디자이너** 반복 개발 : 핵심 기능 구현 후 테스트와 개선을 반복하며 디테일 완성 (Live GDD)

## 3. 텍스트 기반 환경

- **Markdown 기반 기획** : VS Code, Cursor, Obsidian 등 텍스트 환경 활용, LLM 도구 연동 및 데이터 처리에 최적화
- **Git 버전 관리** : 기획서를 코드처럼 관리하여 변경 이력 추적 및 효율적인 협업 가능
- **SSoT (Single Source of Truth)** : 기획서 내 데이터(Code block)를 게임이 직접 로드하여 기획과 구현의 불일치 제거

# Phase 1: 아이디어 발굴

- Pitch 문서 작성 / 빠른 검증
- [피치 문서 작성 가이드라인](010-Game-Pitch-Guide.md)
- [장르 분류 가이드라인](032-Genre-Guide.md)

## 1. 아이디어 수집

- **목적**: 넓게 발산 후 빠르게 수렴, 고유 훅 확보
- **발상법**: Crazy 8s / SCAMPER / 경쟁작 리뷰 분석 / 밈·IP 리믹스
- 아이디어 카드 10~30개 : 2~3문장 + 이미지 1 + 유사작 2
    - "누구/무엇/왜 지금" 한 문장 설명 / 고유 훅

## 2. 피치 문서 작성

- **목적**: 60초 설득력 검증 / 2~3분 슬라이드
- **구성**: 문제/판타지 → 코어루프 3컷 → 차별화 → 타깃/규모 → (개발팀소개)
- **검증**: 내부 5명 중 3명 이상 "해보고 싶다"
- **피치덱 (Pitch Deck)** : 피치 문서를 기반으로 프레젠테이션 작성

# Phase 2: 비전 수립

- 하이 컨셉 기획서 작성 / 방향성 정립
- [용어 쉽게 이해하기](030-Game-Design-Analogies.md)
- [카테고리별 참고 이론](031-Design-Theory-Applications.md)

## 3. 컨셉 기획서

- **목적**: 게임 비전 수립 / 방향성 정립 / 전체 범위 세부 개요 작성
- **구성**:
    - 게임 비전 : 핵심 가치 / Design Pillars 3개 / 차별화 포인트
    - 코어 루프 : 경험 사이클 / 재미 요소 / 감정 곡선 / 리텐션 메커니즘
    - 타깃 전략 : 페르소나 / 플랫폼 / 시장 포지셔닝
    - 경쟁 분석 : 벤치마크 / 차별화 / 레드오션 회피
    - 세계관 & 무드 : 테마 / 톤앤매너 / 감성 방향 / Anti-References
    - 비즈니스 모델 : 수익화 / 라이브 운영 / 성장 전략
    - 성공 정의 : KPI / 킬 크라이테리아 / 개발 범위 & 팀 구성

## 4. 무드보드 & 레퍼런스

- **목적**: 시각·청각적 방향성 구체화 / 팀 내 이미지 싱크 맞추기
- **구성**:
    - 아트 스타일 & 컬러 팔레트 / UI/UX 레퍼런스
    - 게임플레이 레퍼런스 (What/Why) / 사운드/음악 디렉션
    - Anti-References : 하지 말 것

## 5. 기능 사양서

- **목적**: 구현을 위한 구체적 기능 명세 / 기술적 실행 계획 수립
- **구성**:
    - 게임 구조 : 씬 플로우 / 플레이어 설정 / 코어 루프 메커니즘
    - 데이터 개념 : 주요 스탯 / 이벤트 구성 요소 / 조건부 콘텐츠 트리거
    - 제작 방향 : 데이터 생성 접근법 (Python+LLM) / 영상 제작 방식 / 콘텐츠 규모
    - 기술 방향 : 주요 도구 및 플랫폼 / 제작 공정 개요
- **산출물**: 게임 구조 다이어그램 / 스탯/이벤트 개념도 / 제작 파이프라인 / 기술 스택

# Phase 3: 플레이 & 기능 검증

- 프로토타이핑 / 동작 검증, 경험 검증, 재미 검증
- [플레이 & 기능 검증 가이드라인](011-Prototype-Verification.md)

## 6. 스크린 / 플레이 플로우

- **목적**: 사용자 경험의 구조적 설계 / 논리적 흐름과 상태 정의
- **구성**:
    - 스크린 : IA / 화면 상태·전이 / 오류·빈상태 / 퍼널
    - 플레이 : 코어/메타 루프 / 경제·진행·보상 / 이벤트/트리거
- **산출물**: 플로우 다이어그램 / 간단 화면 사양

## 7. 기능 검증 프로토타입

- **목적**: 동작 논리 확인 / 기술적 리스크 조기 제거 / 웹 기반 빠른 검증
- **웹·LLM 간소 구현**: 작은 단위로 원샷 구현
- **버티컬 슬라이스** : 메뉴→로비→게임→결과 필수 화면만
- **구성**:
    - UI/UX 플로우 테스트 : 클릭 경로 / 상태 보존 / 에러 처리
    - 메카닉 로직 검증 : 아이템 획득 조건 / 쿨다운 / 리트라이 규칙 / AI 반응
    - 화면 전환 및 상태 검증: 로딩 / 재시작 / 세션 종료
    - 재미 검증: 작은 단위로 재미 요소 검증
- **산출물**: 웹 데모 링크 / 테스트 체크리스트 / 텔레메트리 이벤트 정의

## 8. GDD 개요 문서

- **목적**: 컨셉기획서 기반 / 구체적 제작 문서 / 지도 역할 (연결 및 개요)
- 비전, 컨셉 항목은 컨셉 기획서 연결, 세부 항목은 세부 문서 연결
- 짧은 서술 + 링크 허브 / 세부는 하위 문서 분리
- 단일 진실 소스 (SSoT) / 계층 3단 이내
- **구성**:
    - **비전/컨셉 문서**: 영감과 방향성을 제시
    - **세계관/월드빌딩**: 배경 세계의 규칙, 역사, 이야기 정의
    - **시스템 문서**: 실행 가능한 구현 사양 (전투, AI, 레벨 등)
    - **경제/밸런스 문서**: 장기 성장 곡선과 자원 흐름 정의
    - **UI/UX 문서**: 사용자 경험의 모든 측면 정의
    - **기술 문서**: 구현 제약과 기술적 결정 명시
    - **검증/로드맵 문서**: 프로젝트 관리와 의사결정 기준

## 9. 세부 문서 (시스템, 데이터, 컨텐츠)

- **목적**: 즉시 구현 가능한 상세 규칙 정의 / 불확실성 제거
- 각 시스템 규칙 상세화 / 일관된 포맷 / 3~10p 권장
- 시스템 설계 문서 : 전투 / 이동 / 퍼즐 / 경제 / 메타진행 등 기능 단위 규칙서
- 데이터 문서 : 캐릭터 / 아이템 / 드랍 / 퀘스트 / 스키마·단위·명명 규칙 + 데이터 시트
- 시스템 <-> 데이터 불일치 0 / 데이터 문서가 진실원본 / GDD 개요는 간단 개요와 링크만
