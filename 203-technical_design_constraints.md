# 기술 설계 & 제약사항 작성 가이드

## 1. 목적 (Purpose)
게임이 반드시 지켜야 하는 기술적 제약(세이브/로드, 성능 예산, 플랫폼 인증, 네트워크 요구)을 설계 단계에서 명문화하여, 출시 직전에 발견되는 구조적 문제를 예방하기 위한 가이드입니다.

## 2. 범위 (Scope)
- **포함**: 세이브/로드 시스템(저장 시점·슬롯·클라우드·버전 마이그레이션), 성능 예산(프레임/메모리/로딩/발열), 플랫폼 인증 요구사항(콘솔 TRC/XR, 모바일 심사), 네트워크 요구(오프라인 모드·재접속), 보안/치트 방지 기본 원칙, 엔진·미들웨어 결정 기록
- **제외**: 서버 인프라 상세 설계, 구체적인 코드 구현, 배포 전략
- **문서 경계**:
  - 데이터 스키마·포맷 정의는 [데이터 스키마 & 코드블록](200-data_schema_codeblocks.md)이 소유합니다. 이 문서는 세이브 데이터의 "버전·마이그레이션 정책"만 다룹니다.
  - 툴·파이프라인·빌드 자동화는 [툴·파이프라인 요구사항](202-tools_pipeline_requirements.md)이 소유합니다.
  - 텔레메트리 이벤트·KPI 정의는 [로그·텔레메트리·KPI](201-analytics_telemetry_kpi.md)가 소유합니다. 이 문서는 수집이 성능 예산을 침범하지 않는지의 관점만 다룹니다.

## 3. 설계 목표 (Design Goals)

1. **신뢰성 (Reliability)**: 플레이어의 진행 상황은 어떤 상황에서도 손실되지 않습니다.
2. **예측 가능성 (Predictability)**: 성능 목표를 수치로 정의하여 최적화 여부를 판정 가능하게 만듭니다.
3. **인증 통과 (Certification-Ready)**: 플랫폼 인증 요구사항을 개발 초기부터 반영하여 재제출 비용을 줄입니다.
4. **우아한 실패 (Graceful Degradation)**: 네트워크 단절, 저장 실패 등 예외 상황에서도 플레이어 경험을 보호합니다.

## 4. GDD 문서 구조 (Structure)

### 4.1 세이브/로드 시스템 (Save/Load System)
- **저장 시점**: 오토세이브 트리거(체크포인트 통과, 방 이동, 주요 선택 직후), 수동 저장 허용 범위, 전투/컷신 중 저장 금지 구간
- **슬롯 정책**: 슬롯 수(1개/다중/무제한), 슬롯당 용량 상한, 백업 슬롯(직전 저장 자동 보존) 여부
- **클라우드 동기화**: 지원 플랫폼(Steam Cloud, PSN, Xbox Live 등), 충돌 해결 규칙(최신 우선 vs 사용자 선택), 크로스플랫폼 세이브 지원 여부
- **버전 마이그레이션**: 세이브 스키마 버전 필드, 구버전 세이브 로드 시 마이그레이션 규칙, 하위 호환 보장 범위(예: 최근 3개 메이저 버전)

세이브 데이터 스키마 버전 정의 템플릿:

```yaml
save_schema:
  version: 3            # 스키마 버전 (마이그레이션 판정 기준)
  min_supported: 1      # 로드 가능한 최소 버전 (이보다 낮으면 로드 거부 + 안내)
  autosave:
    triggers: [checkpoint, area_transition, quest_complete]
    interval_sec: 300   # 트리거 없을 때 최대 저장 간격
    max_write_ms: 50    # 메인 스레드 블로킹 허용치 (세이브 스터터 방지)
  slots:
    manual: 10
    auto: 3             # 순환(rolling) 오토세이브 슬롯
    backup: 1           # 직전 정상 세이브 보존
  migrations:
    - from: 1
      to: 2
      rule: "inventory.gold -> currency.gold 필드 이동"
    - from: 2
      to: 3
      rule: "quest_flags를 bitfield에서 명시적 키-값으로 변환"
```

### 4.2 성능 예산 & 플랫폼 요구사항 (Performance Budget & Platform Requirements)
- **프레임 예산**: 목표 fps와 프레임당 ms 배분(렌더링/게임 로직/물리/UI), 최악 시나리오(최대 동시 유닛 수) 기준 정의
- **메모리·로딩·발열**: 플랫폼별 메모리 상한, 초기 로딩/구간 로딩 목표 시간, 모바일 지속 플레이 시 발열·배터리 목표

| 플랫폼 | 해상도 | 목표 fps | 메모리 상한 | 초기 로딩 |
|---|---|---|---|---|
| PC (권장 사양) | 1440p | 60 | 8 GB | 15초 이내 |
| PS5 / Xbox Series X | 4K(동적) | 60 | 12 GB | 10초 이내 |
| Nintendo Switch | 720p(휴대) / 1080p(거치) | 30 | 3 GB | 20초 이내 |
| Steam Deck | 800p | 40 | 6 GB | 15초 이내 |
| 모바일 (중급기) | 기기 기본 | 30 | 2 GB | 10초 이내 |

- **콘솔 인증 (TRC/XR)**: 플랫폼 홀더별 기술 요구사항 체크리스트 — 저장 중 전원 차단 대응, 컨트롤러 분리 시 일시정지, 시스템 UI 용어 준수, 세이브 데이터 파손 감지·복구
- **모바일 심사**: App Store/Google Play 정책 — 결제 규정, 개인정보 고지, 앱 용량 제한(셀룰러 다운로드 상한), 백그라운드 전환 시 상태 보존

### 4.3 네트워크 & 보안 (Network & Security)
- **오프라인 모드**: 오프라인에서 가능한 기능 범위 명시, 온라인 필수 기능의 실패 시 대체 동작(fallback)
- **재접속**: 세션 중 단절 시 유예 시간, 재접속 시 상태 복원 범위(멀티플레이어는 진행 중 매치 복귀 규칙)
- **치트 방지 기본**: 신뢰 경계 정의(클라이언트는 신뢰하지 않음), 서버 권위(server-authoritative) 대상 데이터 목록, 세이브 파일 변조 대응 수준(싱글플레이는 허용/무시 가능)

### 4.4 선택 및 심화 (Optional & Advanced)
- **엔진·미들웨어 결정 기록 (ADR)**: 엔진 선택 근거, 미들웨어(물리/오디오/네트워크) 채택·탈락 사유, 라이선스 비용·제약을 결정 기록(Architecture Decision Record) 형식으로 보존
- **결정론적 시뮬레이션**: 리플레이·롤백 넷코드·검증 시뮬레이션의 전제 조건 — 고정 소수점/고정 틱레이트 필요 여부를 초기에 결정
- **롤백 넷코드 (개념 수준)**: 입력 지연 방식과 롤백 방식의 트레이드오프, 롤백 프레임 수와 CPU 예산의 관계
- **크로스플랫폼 세이브**: 플랫폼 간 계정 연동 방식, 플랫폼 정책상 이전 불가 데이터(플랫폼 전용 재화 등) 처리

## 5. 관련 개념 (Key Concepts)

### 5.1 세이브/로드
- **Save Stutter (세이브 스터터)**: 저장 시 디스크 I/O가 메인 스레드를 블로킹해 발생하는 프레임 드랍. 비동기 저장으로 예방
- **Autosave vs Manual Save**: 오토세이브는 손실 방지, 수동 저장은 플레이어 통제감 제공. 장르에 따라 조합 설계
- **Save Scumming (세이브 스커밍)**: 저장/로드 반복으로 확률 결과를 재추첨하는 행위. 허용 여부는 디자인 결정 사항 (시드 고정으로 차단 가능)
- **Cross-Platform Save**: 플랫폼 간 세이브 공유. 스키마 통일과 계정 연동이 전제 조건

### 5.2 성능 & 플랫폼
- **Performance Budget**: 프레임 시간·메모리·로딩을 기능별로 배분한 상한. 초과 시 최적화 또는 기능 축소 판정 기준
- **TRC/XR**: 소니(TRC), 마이크로소프트(XR) 등 콘솔 플랫폼 홀더의 기술 인증 요구사항. 위반 시 심사 반려
- **Thermal Throttling (발열 제한)**: 모바일·휴대 기기에서 발열로 성능이 강제 하락하는 현상. 지속 플레이 30분 기준으로 목표 수립

### 5.3 네트워크 & 시뮬레이션
- **Deterministic Simulation**: 동일 입력이 항상 동일 결과를 내는 시뮬레이션. 리플레이·록스텝·검증의 기반
- **Rollback Netcode**: 상대 입력을 예측해 진행하고, 실제 입력 도착 시 되감아 재계산하는 넷코드 방식
- **Server Authority**: 재화·진행도 등 핵심 상태의 판정 권한을 서버가 갖는 구조. 치트 방지의 출발점

## 6. 품질 체크리스트 (Quality Checklist)

### 6.1 세이브/로드
- [ ] 모든 저장 시점과 저장 금지 구간이 정의되었는가?
- [ ] 세이브 스키마에 버전 필드와 마이그레이션 규칙이 있는가?
- [ ] 저장 중 강제 종료(전원 차단) 시 데이터 파손 대응이 정의되었는가?

### 6.2 성능 & 플랫폼
- [ ] 플랫폼별 목표 fps·메모리·로딩 시간이 수치로 정의되었는가?
- [ ] 최악 시나리오(최대 부하 상황) 기준이 명시되었는가?
- [ ] 출시 대상 플랫폼의 인증 요구사항 체크리스트가 확보되었는가?

### 6.3 네트워크 & 보안
- [ ] 오프라인에서 가능한 기능 범위가 명시되었는가?
- [ ] 단절·재접속 시나리오별 동작이 정의되었는가?
- [ ] 서버 권위로 보호할 데이터 목록이 정의되었는가?

## 7. 예시 및 안티패턴 (Examples & Anti-Patterns)

**Case 1: 저장 시점 설계**
- **좋은 예 (Good)**: "체크포인트·지역 이동·퀘스트 완료 시 오토세이브 + 수동 저장 10슬롯 + 직전 세이브 백업 1슬롯, 저장은 비동기 처리로 50ms 이내 완료"
- **안티패턴 (Bad)**: "오토세이브가 보스전 직전 낭떠러지 낙하 중에 동작하여 로드하자마자 사망하는 '데드 세이브' 발생, 백업 슬롯도 없어 진행 불가"

**Case 2: 성능 예산 부재**
- **좋은 예 (Good)**: "프레임 16.6ms 중 렌더링 9ms/로직 4ms/물리 2ms/UI 1ms로 배분, 매주 자동 프로파일링으로 예산 초과 기능을 즉시 식별"
- **안티패턴 (Bad)**: "'최적화는 나중에'라며 수치 목표 없이 개발, 알파 시점에 30fps도 안 나와 콘텐츠를 잘라내는 상황 발생"

**Case 3: Hades - 어디서나 저장 가능한 로그라이크**
- **게임 특성**: 한 판(런)이 30분 이상 걸리는 로그라이크 액션
- **기술 설계**: 언제든 게임을 종료해도 현재 방 기준으로 진행 상황이 보존되어, 재실행 시 런 중간부터 이어서 플레이 가능
- **효과**: "로그라이크는 중간 저장이 없다"는 관성을 깨고 짧은 세션 플레이어도 수용. 세이브 시스템이 접근성과 플랫폼(휴대 모드) 적합성을 동시에 높인 사례

**Case 4: Elden Ring - Steam Deck 검증 대응**
- **게임 특성**: 오픈월드 액션 RPG, PC 출시와 함께 Steam Deck Verified 획득
- **기술 설계**: 휴대 기기의 성능·해상도·컨트롤러 요구사항을 별도 검증 대상으로 관리
- **교훈**: 출시 대상에 휴대형 PC가 포함된다면, 표의 성능 목표에 별도 행으로 정의하고 검증 체크리스트(글꼴 가독성, 기본 컨트롤러 지원)를 GDD 단계에서 확보해야 함

**Case 5: Cyberpunk 2077 - 구세대 콘솔 출시 실패**
- **사건**: 2020년 12월 출시 시 PS4/Xbox One 버전이 심각한 성능 저하와 크래시를 보였고, 소니는 PlayStation Store에서 게임을 한동안 판매 중단하고 환불을 진행
- **원인 관점**: 구세대 콘솔의 메모리·스토리지 제약이 성능 예산으로 명문화·검증되지 않은 채 최신 하드웨어 기준으로 개발이 진행됨
- **교훈**: 가장 낮은 사양의 출시 플랫폼이 성능 예산의 기준선이 되어야 하며, 기준 미달 시 "출시 연기 또는 플랫폼 제외"라는 판정 규칙을 문서에 미리 정의해야 함

**Case 6: 포켓몬 시리즈 - 세이브 슬롯 1개 정책 논쟁**
- **사건**: 포켓몬 본가 시리즈는 전통적으로 세이브 슬롯을 1개만 제공하여, 한 기기를 공유하는 가족·다회차 플레이어의 불만이 지속적으로 제기됨
- **디자인 관점**: 슬롯 1개는 "한 명의 트레이너로서의 여정"이라는 정체성과 세이브 스커밍 억제 효과가 있지만, 기기 공유 환경에서는 접근성 문제가 됨
- **교훈**: 슬롯 수는 단순 기술 사양이 아니라 디자인 의도와 결부된 결정이므로, GDD에 "왜 이 슬롯 정책인가"의 근거를 반드시 기록해야 함

## 8. 핵심 인사이트 (Key Insights)

- **진행 손실은 최악의 버그**: 세이브 데이터 손실은 플레이어 이탈로 직결됩니다. 버전 필드, 백업 슬롯, 파손 감지를 기본으로 설계하세요.
- **수치 없는 성능 목표는 목표가 아니다**: "쾌적하게"가 아니라 "PS5 4K 60fps, 초기 로딩 10초 이내"처럼 판정 가능한 수치로 정의하세요.
- **가장 약한 플랫폼이 기준선**: 출시 대상 중 최저 사양 플랫폼을 성능 예산의 기준으로 삼고, 미달 시의 의사결정 규칙을 미리 정하세요.
- **인증은 나중에 하는 것이 아니다**: TRC/XR·모바일 심사 요구사항은 구조적 결정(저장 방식, 일시정지 처리)에 영향을 주므로 개발 초기에 반영하세요.
- **결정은 기록으로 남긴다**: 엔진·미들웨어·넷코드 방식 같은 되돌리기 어려운 결정은 근거와 함께 결정 기록(ADR)으로 보존하세요.
