# 작성 팁 모음

## 1. 경험 목표 & 문제 정의

### 1.1 문제 중심 접근

- **솔루션이 아니라 문제부터 쓰기**: `"점프 버튼을 두 번 누르면 이단 점프"` → `"플레이어가 수직 공간을 탐색할 방법이 필요함 → 이단 점프로 해결"`.
- **경험 목표 먼저 정의**: `"플레이어가 무엇을 느끼길 원하는가?"`를 한 줄로 쓰고, 그 경험을 위해 기능을 설계한다.
- **스스로에게 재질문**: "내가 진짜 풀고 싶은 문제 한 줄로 말할 수 있어?"에 대답 못하면 아직 쓰지 않는다.

### 1.2 목표 우선 설정

- **경험 목표 → 문제 → 솔루션 순서**: "플레이어 경험 목표" → "현재 문제" → "해결 솔루션" 순으로 항목을 고정한다.
- **디자인 필러 세우기**: 게임을 지탱하는 핵심 키워드 3개(예: 짧은 세션, 명확한 피드백, 공정한 난이도)를 먼저 정하고 모든 기능을 여기에 맞춘다.

## 2. 브레인스토밍 & 레퍼런스 탐색

### 2.1 아이디어 발산

- **바로 타이핑 금지**: 빈 문서를 여는 순간엔 글을 쓰지 말고, 먼저 생각을 밖으로 쏟아내는 단계부터 시작한다.
- **브레인스토밍 수량 목표**: 최소 15개 아이디어를 막 던져본 후에야 "그 중 하나"를 문서에 올린다.
- **다른 매체에서 영감 찾기**: 영화/웹툰/음악에서 비슷한 감정 구조나 연출을 찾아 레퍼런스를 같이 적어둔다.

### 2.2 시각화 & 체화

- **포스트잇 활용**: 아이디어를 문장 대신 키워드로 포스트잇에 쓰고, 벽에서 물리적으로 재배열하면서 생각을 정리한다.
- **스케치 우선**: UI/플로우/전투는 글보다 먼저 박스 그림과 화살표로 대략적인 그림을 그려본다.
- **프로토타입 만져보기**: 실제 입력 장치/장난감/기존 게임을 손으로 만지며 몸으로 상상한 뒤에 글로 정리한다.

## 3. 구조화된 설계 & 표현 방식

### 3.1 기능 상세 설계

- **4요소 균형 쓰기**: 주요 기능은 항상 **메카닉 / 스토리 / 미학 / 기술** 네 줄로 쪼개어 서술한다.
  - **메카닉**: "무슨 규칙인가?"만 담아, 입력/조건/결과를 짧게 수식처럼 적는다.
  - **스토리**: "이 기능이 서사/세계관에서 어떤 의미인가?"를 한 줄로 연결한다.
  - **미학**: 이펙트/사운드/카메라 등 눈과 귀로 느끼는 요소를 간단히 리스트로 적는다.
  - **기술**: 애니/히트박스/네트워크/툴 요구사항 등 구현 포인트를 짧게 명시한다.

### 3.2 문서 구조 설계

- **One-Pager 선행**: 모든 큰 기능은 먼저 한 장짜리로 목표/타겟/코어 루프/차별점만 정리해본다.
- **문서 독자 명시**: 문서 상단에 "이 문서는 누구(직군)를 위한 것인지"를 적고, 그 사람에게 필요한 정보 위주로 쓴다.
- **독자의 시간을 아끼는 글쓰기**: 항상 "요약 → 상세" 순으로 쓰고, 긴 설명 앞에 한 줄 요약을 넣는다.

## 4. 이해관계자 경청 & 정렬

### 4.1 내부 팀 정렬

- **팀에게 먼저 묻기**: "이거 우리 엔진/툴/일정으로 구현 가능해?"를 확인한 뒤에야 기능을 확정한다.
- **게임 전체와의 충돌 확인**: 새 시스템이 기존 밸런스/경제/UI/난이도 곡선과 충돌하지 않는지 먼저 체크한다.

### 4.2 외부 목표 정렬

- **플레이어에게 먼저 묻기**: "우리 타겟 유저가 실제로 원하는 상황/판타지인가?"를 스스로 인터뷰하듯 점검한다.
- **비즈니스 정렬**: 기능이 유지율/과금/바이럴 등 비즈니스 목표 중 무엇을 돕는지 한 줄로 적는다.
- **필러와 안 맞는 기능은 버리기**: 디자인 필러와 연결되지 않는 기능은 "좋아 보여도 기본적으로 넣지 않는다"를 원칙으로 삼는다.

## 5. 검증 필터 & 실험 계획

### 5.1 경험 검증

- **예술적 직관 필터**: "내 감각이 진짜로 멋있다고 느끼는가?"를 묻고, 애매하면 미학/연출을 더 다듬는다.
- **인구통계 필터**: 타겟 연령/성별/지역을 떠올리며 "이 사람들이 실제로 이해하고 즐길 수 있을까?"를 체크한다.
- **경험 디자인 필터**: 기능 설명 뒤에 "그래서 플레이어는 어떤 경험을 한다?"라는 문장을 반드시 붙인다.

### 5.2 차별화 검증

- **혁신 필터**: "이게 다른 게임 대비 뭐가 다르지?"를 한 줄로 못 쓰면 기능을 줄이거나 비틀어서 차별점을 만든다.
- **비즈니스 필터**: "이 기능은 어떤 마케팅 문장으로 홍보할 수 있지?"를 떠올려 보고, 문장으로 안 뽑히면 재검토한다.

### 5.3 실행 가능성 검증

- **엔지니어링 필터**: 구현 난이도/성능/네트워크 리스크를 적어보고, 최소 한 개 대안을 함께 적는다.
- **커뮤니티/사회 필터**: 문화적 이슈, 독성, 과도한 자극 요소가 없는지, 커뮤니티에 어떤 행동을 유도할지 적어본다.
- **리스크와 대안 동시 기록**: 요구사항을 적을 때, 바로 아래에 "리스크"와 "플랜 B"를 함께 적는다.

### 5.4 검증 방법론

- **플레이테스트 필터**: "이걸 테스트한다면 어떤 지표와 피드백으로 판단할까?"를 미리 정리해둔다.
- **실험 계획까지 포함**: 기능 설명 뒤에는 "이걸 어떻게 실험/검증할지(테스트 방식과 지표)" 한 줄을 붙인다.

## 6. 문서 운영 & 버전 관리

- **버전/변경 이력 관리**: 문서 맨 위에 버전과 변경 요약을 적어, GDD를 ‘살아있는 설계도’로 다룬다.
- **문서 운영 관점**: GDD를 ‘한 번 쓰고 버리는 문서’가 아니라 계속 업데이트되는 레퍼런스로 취급한다.
- **검색성을 고려한 작성**: 검색과 재사용을 위해 제목/소제목/태그/요약을 신경 써서 작성한다.
